# 战斗系统优化需求（Godot 版）

> 文档版本：V1.5  
> 编写日期：2026-07-14  
> 状态：高质量 2.5D 大战场、敌左我右横向布局、局部树林/坡顶/浅滩、实时接敌、基础 RTS 指挥、攻城内外分区、总兵力条与倒计时已实现；技能军令、增援和撤退深化保留为后续切片
> 适用工程：`godot/` 下的 Godot 4.7 .NET / C# 正式版本

## 1. 文档目的

本需求用于重构当前“月末直接比较攻守数值并生成文字战报”的战斗流程，使战斗具备以下能力：

1. 军团人数、兵种构成、训练、士气、疲劳、粮草、地形和战术共同决定战局。
2. 主将和所有随军副将都必须对战局产生可计算、可展示、可复盘的实际影响。
3. 玩家军团参战时进入独立战斗画面，完成排兵布阵后再开始结算和演出。
4. 野战使用不同地形背景和五兵种独立动画，损耗随战斗阶段实时变化。
5. 攻城战具有外城墙、城门、内城三层过程，不再一次比较后直接宣布城池易主。
6. 战前预估、正式结算、动画时间线和战后报告使用同一份结果数据，确保结果可解释、可复现。
7. 一支战略军团必须按真实人数展开为大量战斗队，形成前军、中军、后军和左右翼，而不是始终只显示 3 个大图标。
8. 各兵种必须遵守阵位、射程和行动规则；例如后排弓兵只推进到有效射程，随后停驻齐射，不会无条件走到中线肉搏。

本文是战斗专项的详细需求来源。若与 `docs/01-requirements.md` 中较早的概述冲突，以本文为准。

## 2. 改造前实现审计

> 本节保留 V1.1 改造前基线，便于追溯问题来源。当前固定步长实时结算、十二战术、单一主战术、阵型军令和结构化战报实现以 `BattleSystem.cs` 与 `docs/02-implementation.md` 为准。

### 2.1 当前军团规则

- 出征兵力最低为 1,000。
- 出发城必须至少保留 3,000 守军。
- 界面允许选择 1 名主将和最多 2 名副将。
- `ArmyData.Composition` 已能保存兵种字典，但当前出征界面会把全部兵力编成同一种兵种。
- 军团记录兵力、军粮、倾向、主战术、路线和剩余行军日数。
- 当前界面允许输入最高 100,000 人，但没有根据主将统率、城市人口或后勤设置实际统兵上限。

### 2.2 当前战斗公式

当前攻城攻击力为：

```text
副将加成 = Σ((副将统率 + 副将智力) / 500)
兵种加成 = 骑兵占比 × 0.08 + 攻城器械占比 × 0.14

攻击力 = 兵力
       × (0.65 + 主将统率 / 150 + 副将加成 + 兵种加成)
       × 作战倾向

防御力 = 守军
       × (0.75 + 城防 / 130 + 训练 / 300)

胜利条件 = 攻击力 > 防御力 × [0.88, 1.12] 的确定性随机值
```

当前损失采用固定比例：

| 结果 | 攻方损失 | 守方损失 |
| --- | ---: | ---: |
| 攻方胜利 | 18% | 48% |
| 攻方失败 | 38% | 22% |

### 2.3 当前问题

| 模块 | 当前情况 | 主要问题 |
| --- | --- | --- |
| 主将 | 只有统率直接进入攻击公式 | 武力、智力、魅力、特性和临场职责没有明确作用 |
| 副将 | 每人仅贡献“统率＋智力”的统一加成 | 每名副将没有独立职责、事件、动画和贡献记录 |
| 守城武将 | 防御公式只读取城池数据 | 太守和城中武将不影响守城结果 |
| 兵种 | 只有骑兵和攻城器械产生简单正加成 | 步、枪、弓没有独立价值，也没有完整克制链 |
| 战术 | 选择后写入军团和战报 | 战术没有参与胜负计算，也没有明确的兵种条件提示 |
| 地形 | 道路数据包含地形 | 战斗结算和战斗画面没有真正使用地形 |
| 损耗 | 按胜负套固定比例 | 无法解释接战过程，也不能按兵种产生不同伤亡 |
| 攻城 | 到达后立即进行一次总评分 | 没有城墙、城门、内城和持续围城状态 |
| 战斗画面 | 固定平原背景、少量大编组标记加文字战报 | 每侧只有约 3 个大块，没有人数展开、前后排、射程、实时损耗和多武将表现 |
| 战报 | 仅保存主将、兵力、损失和胜负 | 无地形、编制、阶段、武将贡献、计策触发和城防过程 |

### 2.4 可直接复用的现有资产

- 平原、丘陵、水路、山地 4 套野战背景。
- 中原、西凉、江南、南蛮 4 套分层攻城背景。
- 步兵、枪兵、弓兵、骑兵、攻城器械 5 套 4×2、8 帧透明图集。
- 112 名剧本武将的全身肖像。
- 33 座城池已有地域家族和地图图标映射，可用于选择对应城堡背景。

## 3. 设计目标与边界

### 3.1 核心目标

- **人数重要**：兵力优势必须有意义，不能靠单个武将无限放大少量兵力。
- **克制可信**：步兵不克制骑兵；枪兵负责拒止骑兵，骑兵克制步兵和弓兵，但会被枪阵压制。
- **全员参战**：玩家选择的每一名出战武将都必须有明确职责，并至少改变一个正式结算字段。
- **过程真实**：动画展示的推进、交锋、破城和损耗必须来自正式战斗时间线，而非纯装饰。
- **实时指挥**：玩家以战斗队为单位点选、框选、移动、集火和固守，在自动接敌基础上主动改变局部战局。
- **规模可信**：军团人数越多，展开的战斗队越多；同样兵种也要形成多列、多排和多个战术分队，而不是合并成一个兵种图标。
- **攻城分层**：攻城必须先处理墙体或城门，再进入内城与守军决战。
- **规则同源**：玩家、AI、战前预估、自动结算和战斗画面必须使用同一套规则。
- **信息可解释**：战报能够回答“为什么赢”“为什么这个副将有用”“城门为什么没有攻破”。

### 3.2 明确不做

- 不做逐个表现士兵的选择、寻路和碰撞；RTS 操作对象始终是战斗队。
- 不要求画面实际渲染几千或几万名士兵。
- 不做独立的武将单挑小游戏。
- 不允许玩家在动画已经结算后通过临时切换选项改变既定结果。
- 不允许 AI 使用玩家不可见的隐藏路线或隐藏命令作弊。

## 4. 战斗触发与整体流程

### 4.1 战斗类型

首版优化覆盖：

1. 道路遭遇战。
2. 城外野战。
3. 关隘战。
4. 攻城战。
5. 突围战和追击战作为结果阶段，不单独制作新场景。

### 4.2 玩家战斗流程

```text
军团发生接战
→ 创建并保存 PendingBattle
→ 暂停当月军事结算
→ 进入战前军议
→ 安排阵型、武将职责和主战术
→ 玩家点击“开始战斗”
→ 领域层以固定步长实时计算移动、寻敌、攻击与损耗
→ 玩家点选或框选战斗队，下达移动、集火、固守等即时命令
→ 表现层同步读取当前状态和持续产生的 BattleTimeline 事件
→ 写入最终状态、战报、关系变化和历史年表
→ 返回天下地图并继续剩余月度结算
```

玩家参与的战斗不得在打开战斗画面前完成最终结算。战斗页面必须从“战前军议、进度 0%”开始。

### 4.3 AI 对 AI 战斗

- AI 对 AI 使用相同战斗上下文和计算器，但可以不打开表现画面。
- 系统仍需生成完整阶段结果和武将贡献，保证玩家之后查看战报时信息一致。
- 自动演进模式可以跳过动画，但不得使用简化胜负公式。

### 4.4 跳过和降低动态效果

- `×1`：完整播放，目标时长 14–20 秒。
- `×2`：目标时长 7–10 秒，只改变播放速度，不重新计算结果。
- “跳过演算”：立即消费同一条时间线并展示结果，不重新抽取随机值。
- 降低动态效果：减少位移、震动和粒子，不得把整场战斗压缩为一秒结束。

## 5. 军团人数与编制

### 5.1 军团人数范围

- 最低出征人数：1,000。
- 数值输入步长：100。
- 单个兵种最小有效编组：500；少于 500 时不能单独占用一个辅助兵种位。
- 单军团硬上限：25,000。
- 出发城最低留守兵力：`max(3,000, 当前驻军 × 25%)`。

### 5.2 统兵容量

军团可用上限由主副将共同决定：

```text
统兵容量 = 3,000
         + 主将统率 × 120
         + Σ(副将统率 × 30)

最终容量 = clamp(统兵容量, 5,000, 25,000)
```

- 超过统兵容量时禁止确认出征，不采用隐藏惩罚。
- 出征界面必须显示“当前兵力 / 统兵容量”，并列出主将和每名副将提供的容量。
- 守城没有 25,000 的驻军上限，但超过守城主将指挥容量的部分只能作为后备队，不能同时投入一个战斗阶段。

示例：主将统率 90，两名副将统率 80、70，则容量为 `3,000 + 10,800 + 2,400 + 2,100 = 18,300`。

### 5.3 军团规模与战斗队展开

为避免“战略军团”“画面军团”概念混淆，统一使用以下术语：

- **战略军团**：地图上由主将率领的一支 `ArmyData`，总兵力为 1,000–25,000。
- **战斗队**：战略军团进入战斗画面后展开的战术分队，是正式计算和动画的最小队形单位。
- **表现士兵**：战斗队内部用于制造规模感的 3–6 个士兵动画，不逐人参与寻路和计算。

战斗队数量按军团人数动态生成：

```text
理论战斗队数 = ceil(军团现役兵力 / 600)
正式战斗队数 = clamp(理论战斗队数, 1, 40)
```

| 实际兵力 | 规模名称 | 参考战斗队数量 | 单队参考兵力 |
| ---: | --- | ---: | ---: |
| 1–3,999 | 小军 | 1–7 队 | 约 400–600 |
| 4,000–9,999 | 标准军团 | 7–17 队 | 约 570–600 |
| 10,000–17,999 | 大军团 | 17–30 队 | 约 590–600 |
| 18,000–25,000 | 主力军团 | 30–40 队 | 约 600–625 |

- 6,000 人军团约展开 10 个战斗队，12,000 人约 20 个，24,000 人约 40 个。
- 低于 600 人时只展开 1 个战斗队，避免数百人与数千人在画面上拥有相同队伍数量。
- 兵力按兵种占比分配战斗队；当总队数少于有效兵种数时，优先保留人数最多的兵种，不额外虚增战斗队。
- 同兵种的多个战斗队必须分布为多排、多列或左右分队，禁止叠在同一坐标伪装成一个大图标。
- 每个战斗队绑定兵种、代表兵力、当前兵力、士气、阵位、逻辑坐标、射程、目标和行动状态。
- 每个战斗队内部绘制 3–6 名表现士兵；因此 20 个战斗队可呈现约 60–120 名士兵，但只计算 20 个战斗队。
- 损失先写入战斗队代表兵力，再表现为队列变稀、旗帜受损、后撤、溃散或退出战场。
- 每侧首版上限为 40 个正式战斗队；超过性能预算时优先减少单队表现士兵，不得把 40 队重新压回 3 个大图标。

### 5.4 混合兵种

- 一支军团包含 1 个主兵种和最多 2 个辅助兵种。
- 各兵种人数之和必须等于军团总兵力。
- 攻城器械只能作为辅助兵种，不能成为唯一兵种或主兵种。
- 主兵种不得低于军团总兵力的 40%。
- 单个辅助兵种不得低于 500 人，也不得低于总兵力的 10%。
- 战损、伤兵、溃兵和补员按兵种分别保存，战后不能重新随意改写幸存编制。

### 5.5 军团持续状态

每支军团新增或正式使用以下状态：

- 分兵种现役人数。
- 分兵种伤兵人数。
- 溃兵人数。
- 全军训练 0–100。
- 全军士气 0–100。
- 疲劳 0–100。
- 携行军粮和预计可用天数。
- 当前阵型。
- 主战术及其兵种条件状态。
- 主将与副将职责分配。
- 当前围城层级和城防破坏状态。

## 6. 武将作用与全员影响规则

### 6.1 出战武将数量

优化首版继续采用：

- 1 名主将。
- 最多 2 名副将。
- 守城方同样使用 1 名守城主将和最多 2 名守城副将。

以后如扩展到更多部将，继续复用职责系统，不重新设计另一套加成规则。

### 6.2 “所有武将都影响战局”验收契约

每一名被选入战斗的武将都必须同时满足：

1. 在战前界面拥有独立肖像、职责和贡献预览。
2. 至少改变一个正式结算字段，例如局部战力、计策效果、损耗、士气、结构伤害、撤退或伤兵回收。
3. 在战斗时间线中至少拥有一条可追踪记录；没有触发特殊事件时也要记录基础职责贡献。
4. 在战后报告中显示其数值贡献和关键行为。

禁止出现“武将被带出征，但只在名单或肖像中出现，不参与公式”的情况。

### 6.3 五维能力的战斗用途

| 能力 | 战斗用途 |
| --- | --- |
| 统率 | 统兵容量、阵型效率、部队执行命令、全军损耗控制、撤退秩序 |
| 武力 | 前锋冲击、近战突破、追击、保护阵线、降低所属战斗队接战损失 |
| 智力 | 计策强度、识破敌策、夜袭火攻、攻城方案和战场情报 |
| 政治 | 军需组织、围城后勤、伤兵收容、城内动员、攻占后的秩序与缴获控制 |
| 魅力 | 初始士气、士气恢复、溃兵收拢、部队忠诚、低士气时维持军令 |

政治不会直接提高野战杀伤，但在后军、围城和战后阶段必须有实际价值。

### 6.4 主将

主将评分：

```text
主将原始评分 = 统率 × 0.65
             + 智力 × 0.15
             + 武力 × 0.10
             + 魅力 × 0.10

主将有效评分 = 主将原始评分
             × 健康修正
             × 疲劳修正
             × 协同修正

主将倍率 = clamp(0.90 + 主将有效评分 × 0.0025, 0.90, 1.15)
```

主将倍率作用于全军，但主将仍需承担以下具体职责：

- 决定统兵容量。
- 决定阵型基础完成度。
- 决定全军初始士气上限和失利时的士气衰减。
- 决定主动撤退是否有序。
- 在计策冲突时提供军令执行能力。

### 6.5 副将职责

副将必须被分配到一个职责。系统可根据属性推荐，但玩家可以更换。

| 职责 | 属性权重 | 主要作用 | 主要阶段 |
| --- | --- | --- | --- |
| 前锋 | 武力 55%、统率 35%、魅力 10% | 提高前军冲击，减少首次接战损失，增强追击 | 推进、接战、决胜 |
| 军师 | 智力 65%、统率 20%、政治 15% | 提高计策强度、识破与主战术执行 | 全阶段 |
| 侧翼 | 统率 45%、武力 35%、智力 20% | 提高骑兵迂回、弓兵换位和合围完成度 | 推进、接战 |
| 后军 | 统率 35%、智力 25%、政治 20%、魅力 20% | 维持补给、收拢溃兵、降低撤退和断粮损失 | 决胜、撤退 |
| 攻城督 | 智力 40%、统率 30%、武力 20%、政治 10% | 提高攻城器械效率、城门伤害和强攻秩序 | 攻城专用 |

副将职责加成为：

```text
职责原始评分 = 对应五维加权结果
职责有效评分 = 原始评分 × 健康修正 × 疲劳修正 × 协同修正
局部职责加成 = 职责有效评分 / 100 × 8%
```

- 局部职责加成只作用于相应战斗队或阶段，不直接叠加到全部兵力。
- 单名副将的基础职责贡献上限为 8%。
- 单纯因为属性较低，携带副将不得比“不带副将”更差；只有受伤、严重疲劳、关系冲突或违令事件可以产生负面影响。

### 6.6 健康、疲劳与协同

```text
健康修正 = 0.75 + 健康 / 400          # 0.81–1.00
疲劳修正 = 1.00 - 疲劳 × 0.0025       # 0.75–1.00
协同修正 = 0.96–1.04
```

- 协同来自主副将关系、共同作战次数、性格冲突和近期争功事件。
- 当前 Godot 数据尚未保存武将关系时，首版可先使用共同作战次数、特性与理念相合度生成协同。
- 后续接入正式关系网络后，不更改主公式，只替换协同来源。

### 6.7 武将特性

- `Traits` 必须通过数据表映射为条件效果，不在战斗代码中按武将 ID 写特殊分支。
- 特性效果必须指定：触发战斗类型、阶段、适用兵种、数值、上限和战报文本。
- 单个特性建议影响 2%–6%，同一阶段全部武将特性总影响不得超过 15%。
- 高武力武将可以触发“突破、断后、护阵”等军团事件，但不进入独立单挑界面。
- 高智力武将可以触发“识破、诱敌、火攻、断粮”等计策事件。

### 6.8 守城武将

- 守城主将默认从太守和城内可用武将中选择统率最高者，玩家有预警时间时可以调整。
- 太守不是最佳统帅时仍参与后勤或内城职责，不得完全消失。
- 守方最多选择两名副将，职责可为城墙守备、城门督战、军师、预备队或突围指挥。
- 没有可用守城武将时，驻军按“无将指挥”处理，阵型效率和士气上限降低，但仍可作战。

## 8. 阵型与作战倾向

### 8.1 阵型

| 阵型 | 适用 | 正面效果 | 代价/被克制 |
| --- | --- | --- | --- |
| 雁行阵 | 步弓混编 | 弓兵射界和全线稳定提高 | 被高速侧击时后排风险增加 |
| 锋矢阵 | 骑兵、精锐步兵 | 首次冲击提高 8% | 远程损失和两翼风险提高 6% |
| 鹤翼阵 | 有侧翼兵力的混编 | 合围完成时决胜提高 8% | 中军兵力不足时容易被突破 |
| 盾阵 | 步兵、枪兵 | 远程与骑兵冲击损失降低 10% | 推进速度和追击效果降低 8% |
| 攻城阵 | 步兵＋攻城器械 | 器械保护和结构伤害提高 | 野战机动和侧翼防护降低 |

- 阵型不是纯视觉选项，必须进入阶段计算。
- 阵型完成度受主将统率、军团训练、地形和疲劳影响。
- 编制不满足阵型条件时可以选择，但界面必须显示效果折损。

### 8.2 两层布阵模型

布阵分为“全军阵型”和“兵种队形”两层。全军阵型决定整支军团的轮廓与总体加成，兵种队形决定各战斗队站在哪里、何时停止和如何作战。

战前布阵盘采用 **3 个纵深 × 5 条战线**：

| 纵深 / 战线 | 左翼 | 左中 | 中军 | 右中 | 右翼 |
| --- | --- | --- | --- | --- | --- |
| 前军 | 正面迎敌、拒止、先锋 | 正面迎敌 | 正面主战 | 正面迎敌 | 正面迎敌、拒止、先锋 |
| 中军 | 侧翼支援 | 增援与预备 | 主将、预备队 | 增援与预备 | 侧翼支援 |
| 后军 | 骑兵待机或弓兵 | 弓兵 | 弓兵、器械 | 弓兵 | 骑兵待机或弓兵 |

- 玩家拖放的是“某兵种部署区域”或“军令块”，不需要逐个摆放 20–40 个战斗队。
- 系统按人数和兵种占比，把该兵种自动展开为多个战斗队并填充到所选区域。
- 玩家可以调整兵种的纵深、战线、队形、目标优先级和进退军令；点击任意区域可预览实际展开效果。
- 每个阵位都有容量。超出容量时战斗队向相邻战线展开；仍无法容纳时形成过密阵列，移动降低 8%、承受远程损失提高 8%。
- 自动布阵必须先生成一套可用方案，例如步枪前排、弓兵后排、骑兵两翼、器械后军；玩家可以直接采用或修改。
- 阵位不是视觉标签，必须影响射程、掩护、侧击、接敌顺序、目标选择和损耗。

总战斗队数：ceil(12,000 / 600) = 20
步兵：10 队
枪兵： 3 队
弓兵： 5 队
骑兵： 2 队

采用“雁行阵＋稳健”的自动布阵建议：

| 阵位 | 展开 |
| --- | --- |
| 前军左中、中军、右中 | 6 队盾墙步兵＋3 队密集枪兵，组成连续接触线 |

## 9. 战术系统

### 9.1 战术数据要求

每个战术必须数据化保存：

- 适用战斗类型。
- 触发阶段。
- 偏好兵种和阵型。
- 基础效果与风险。
- 智力、统率或武力权重。
- 被哪些战术克制。
- 失败、成功和被识破的时间线文本。
- 主战术所需的兵种与最低占比条件。

### 9.2 首批十二战术

| 战术 | 适用阶段 | 核心作用 | 主要风险 |
| --- | --- | --- | --- |
| 稳步推进 | 推进、接战 | 小幅提高阵型稳定和命令执行 | 爆发力低 |
| 盾墙固守 | 推进、接战 | 降低弓兵和骑兵造成的损失 | 推进与追击下降 |
| 佯退诱敌 | 接战、决胜 | 诱使敌阵前移后反击 | 被识破时士气受损 |
| 夜袭 | 战前、推进 | 降低敌方士气和补给 | 疲劳高或情报差时失败 |
| 火攻 | 推进、攻城 | 造成士气、营寨或城门额外伤害 | 受天气、地形和智力影响 |
| 合围 | 接战、决胜 | 提高侧翼和追击效果 | 中军兵力不足时被突破 |
| 箭雨 | 推进、接战 | 弓兵集中压制前排 | 弓兵暴露，怕骑兵冲击 |
| 骑兵突击 | 接战 | 对弓兵和已动摇阵线造成高冲击 | 遇枪阵、盾阵损失增加 |
| 坚营 | 战前、接战 | 提高防守和士气恢复 | 主动输出降低 |
| 断粮道 | 战前、决胜 | 降低敌方补给与持续作战能力 | 需要情报和机动兵力 |
| 云梯攻城 | 城墙阶段 | 提高对城墙的同步进攻效率 | 攻方步兵伤亡较高 |
| 掘地坏墙 | 围城、城墙阶段 | 稳定削弱城防和墙体 | 用时长，怕突围和反制 |

### 9.3 战术结算

- 主将负责军令执行，军师负责计策强度和识破，相关职责副将必须分别进入公式。
- 战术最终影响建议控制在 −12% 至 +15%，避免单次计策覆盖全部人数和编制差距。
- 战术成功不是独立随机按钮，结果由战术基础值、执行武将、敌方识破、情报、地形和确定性波动共同决定。
- 主战术的兵种条件在出征和开战前校验；条件不足时显示明确要求并禁止继续，战中不自动更换战术。

## 10. 战斗计算模型

### 10.1 分阶段计算原则

战斗不再只计算一个总分，而是按阶段依次计算。每个阶段读取上阶段剩余兵力、士气、阵型和结构状态。

野战阶段：

1. 列阵。
2. 推进与远程压制。
3. 正面接战与侧翼行动。
4. 决胜、追击或撤退。

### 10.2 逻辑战场、射程与自动行动

战斗表现不使用物理碰撞和逐兵寻路，而使用确定性的逻辑战场：

- 横向交战轴为 `0–1000`，敌军出生于左侧、玩家出生于右侧；纵向使用 5 条战线。
- 战斗队只保存逻辑坐标、目标阵位和状态，画面再把逻辑坐标映射为像素坐标，因此分辨率不会改变战果。
- 同阵营战斗队可以轻微视觉错位，但正式队形锚点不得互相穿插；换线通过预设通道完成。
- 双方不需要发生碰撞才算接战，只要进入兵种有效攻击距离且满足射界，就生成正式攻击事件。

首版距离参数如下，后续统一由兵种配置表调整：

| 兵种/行为 | 有效距离 | 说明 |
| --- | ---: | --- |
| 步兵近战 | ≤40 | 接敌后停止推进并进入格挡、挥砍循环 |
| 枪兵近战 | ≤55 | 比步兵略早接敌；对正面目标有效 |
| 枪兵架枪侦测 | 140 | 骑兵进入该距离且正在冲锋时，枪兵停步架枪 |
| 弓兵最佳射程 | 120–280 | 目标进入 280 后停止前进并射击 |
| 弓兵极限射程 | 300 | 281–300 命中和伤害降低；超过 300 必须继续移动或换目标 |
| 弓兵危险距离 | <80 | 无前排掩护时优先后撤；无法后撤则近战效率大幅下降 |
| 骑兵冲锋准备 | 120–240 | 距离足够且通道可用时加速冲锋 |
| 骑兵近战 | ≤45 | 完成冲击后进入短暂缠斗，再按军令脱离或持续作战 |
| 攻城器械对结构 | ≤35 | 指冲车等近距器械；远程器械后续独立配置 |

每个战斗队使用统一状态机：

```text
待命 Hold
→ 前进至攻击距离 AdvanceToRange
→ 攻击 Attack
→ 动作恢复 Recover
→ 重整或换位 Reposition
→ 追击 Pursue / 后撤 Retreat
→ 溃散 Rout
```

弓兵必须遵循以下规则：

1. 默认位于后军，跟随本战线前排移动，并保持 90–140 的支援间隔。
2. 可攻击目标进入有效射程后立即停止前进，在原地完成搭箭、拉弓、箭矢飞行和结算事件。
3. 目标退出极限射程时才继续前进或换目标，不得因为全军推进命令持续走入肉搏区。
4. 本战线前排仍完整时，弓兵不得越过前排；前排溃散后按“坚守、后撤、向中军靠拢”军令处理。
5. 友军前排存在时允许抛射，但命中修正 −8%；同线直接射界被友军完全遮挡时，必须改用抛射、换线或换目标。
6. 箭矢到达目标的时间点才应用损耗，不能在弓兵抬手时提前扣血。

其他兵种自动行动规则：

- 步兵推进到近战距离后形成接触线，优先保持阵线并为后排提供掩护。
- 枪兵检测到骑兵冲锋时中断普通推进并架枪；若侧后方遭冲击，克制效果显著降低。
- 骑兵优先走左右翼预设弧线，避开完整枪阵；冲击后根据军令脱离整队或持续追杀。
- 攻城器械只沿受保护的通道向城墙或城门推进，不会在野战部队前方自行冲锋。
- 目标选择由战线、距离和战前优先级共同决定。玩家只设置“优先前军、优先远程、优先骑兵、优先器械”等军令，不做即时点杀。

### 10.3 战斗队战力公式

```text
兵种基础战力 = 当前现役人数 × 兵种野战系数

战斗队阶段战力 = 兵种基础战力
             × 训练修正
             × 士气修正
             × 补给修正
             × 疲劳修正
             × 地形修正
             × 兵种对位修正
             × 阵型修正
             × 阵位与射程修正
             × 正面/侧翼/背击修正
             × 掩护与暴露修正
             × 对应副将职责修正

全军阶段战力 = Σ(当前阶段能有效攻击的战斗队阶段战力)
             × 主将倍率
             × 战术修正
             × 作战倾向修正
             × 守方/结构修正
             × 确定性阶段波动
```

- 只有进入有效射程、拥有合法目标且未处于溃散/恢复状态的战斗队才能在该时间片造成伤害。
- 后排弓兵的输出不能因为“全军战力求和”而无视距离提前生效。
- 侧翼攻击正面完整的目标时获得小幅夹击加成；从目标背面完成攻击时获得更高士气打击。
- 前排完整时，后排承受的正面近战伤害显著降低；前排形成缺口后，后排暴露并成为骑兵优先目标。
- 同一目标周围可有效接战的队数受展开宽度限制，禁止 20 个战斗队在同一时间无损耗地围攻一个点。

### 10.4 部队状态修正

```text
训练修正 = 0.80 + 训练 / 500          # 0.80–1.00
士气修正 = 0.75 + 士气 / 400          # 0.75–1.00
补给修正 = 0.70 + 补给充足度 × 0.30   # 0.70–1.00
疲劳修正 = 1.00 - 疲劳 × 0.003        # 0.70–1.00
```

- 部队状态主要体现惩罚，不让满状态再与其他加成重复膨胀。
- 补给充足度按剩余可战天数计算，不能只判断“粮草是否为零”。
- 各大类乘区分别设置上下限，最终全军倍率不得无上限叠加。

### 10.5 确定性波动

- 每个阶段从存档随机状态生成一次 0.96–1.04 的波动。
- 相同存档、相同命令、相同职责和相同随机状态必须得到完全一致的时间线和结果。
- 不再使用一次 ±12% 随机阈值直接翻转整场胜负。

### 10.6 结果等级

最终战力比用于生成结果倾向，但还需结合士气和撤退状态：

| 战力比 | 结果倾向 |
| ---: | --- |
| ≥1.45 | 大胜 |
| 1.15–1.45 | 胜利 |
| 0.90–1.15 | 胶着或小胜负 |
| 0.70–0.90 | 失败 |
| <0.70 | 大败 |

城池攻防不能只凭最终战力比判定易主，还必须满足攻城胜利条件。

## 11. 损耗、士气与撤退

### 11.1 实时损耗

每个有交战行为的阶段都生成双方损耗：

```text
阶段损耗 = 暴露兵力
         × 阶段基础损耗率
         × 敌我阶段战力比修正
         × 兵种易损修正
         × 阵型与掩护修正
         × 作战倾向修正
```

建议阶段基础损耗范围：

| 阶段 | 单阶段基础损耗 |
| --- | ---: |
| 推进与远程压制 | 0.5%–2% |
| 正面接战 | 4%–12% |
| 决胜 | 2%–10% |
| 溃败追击 | 3%–15% |

- 动画中的损耗数字必须直接读取阶段损耗事件。
- 总损失按兵种和战斗队分配，不能战后再按比例伪造。
- 已经减员的战斗队在后续阶段使用剩余人数继续计算。

### 11.2 损失分类

战斗损失拆分为：

- 阵亡/永久损失。
- 伤兵，可在有医疗和补给的城市逐月恢复。
- 溃兵，战后可能被主将或后军副将收拢。
- 被俘，主要来自溃败、撤退路线被切断和城池陷落。

不同结果的拆分比例由战斗阶段决定，不使用固定全局比例。追击阶段优先增加溃兵和被俘，强攻城墙优先增加伤兵和永久损失。

### 11.3 士气

士气变化来源：

- 武将魅力和主将统率。
- 兵种克制是否成功发挥。
- 阵型是否被突破。
- 计策成功或被识破。
- 友军战斗队溃散。
- 城墙、城门是否失守。
- 补给不足和撤退路线状态。

士气低于 25 时战斗队可能后撤，低于 10 时进入溃散。高统率主将、后军副将和高魅力武将可以收拢，但不能无限恢复。

### 11.4 撤退

- 主动撤退：玩家或条件军令触发，损失较低。
- 被迫撤退：士气或战力崩溃触发，增加溃兵。
- 无合法撤退道路：增加被俘和军备损失。
- 前锋负责断后时可以降低全军撤退损失，但自身负责的战斗队承受更多风险。

## 12. 攻城战

### 12.1 攻城不是一次总分比较

攻城分为：

1. 围城准备。
2. 外城墙/城门阶段。
3. 缺口争夺阶段。
4. 内城决战阶段。
5. 占领、撤围或继续围城。

每层分别保存状态，未突破时不能直接进入下一层。

### 12.2 三层城防

| 层级 | 正式状态 | 主要攻击方式 | 主要守方作用 |
| --- | --- | --- | --- |
| 外城墙 | 耐久 0–100 | 云梯、射击压制、掘地、器械破坏 | 墙头弓兵、守城主将、城防设施 |
| 城门 | 耐久 0–100 | 冲车、火攻、强攻 | 城门督战、枪兵堵口、预备队 |
| 内城 | 控制度 0–100、守军士气 | 巷战、合围、劝降 | 太守、内城预备队、民心与治安 |

- 外城墙和城门至少突破一个有效入口后，攻击军才能大规模进入缺口阶段。
- 占领城池需要：形成有效缺口，并且守军士气低于 25 或内城防线被击败。
- 单纯攻击方总战力较高但没有突破入口时，只能取得围城优势，不能直接易主。

### 12.3 攻城阵位与大量战斗队

攻城沿用“战略军团展开为大量战斗队”的规则，但 3×5 布阵盘改为攻城语义：

- 前军对应护城壕、城墙根部和城门前，主要由盾兵、枪兵护卫、云梯队和冲车队占用。
- 中军对应弓弩压制线和增援线，负责向缺口补充战斗队。
- 后军对应器械集结、主将和预备队；没有护卫的器械会成为守军出城突袭的优先目标。
- 5 条战线对应多个城墙段和城门通道。集中攻门会让更多战斗队排队进入窄通道，结构伤害集中，但同一时间能接战的队数受限。
- 多点登城会把战斗队分配到左右墙段，展开面更宽，但每个突破点的兵力和武将支援被分散。
- 城门或墙体形成缺口后，只允许与缺口宽度相符的战斗队依次进入；后续队伍在入口外等待，不能把全军战力瞬间灌入内城。
- 守军弓兵部署在墙头射击位，目标进入射程后停驻射击；攻方弓兵进入压制射程后停止，为云梯和器械提供掩护。
- 内城阶段重新生成街巷战线与纵深阵位，幸存战斗队按入口顺序展开，不得直接瞬移成三组对撞。

### 12.4 城防修正

```text
城防减伤 = 1 + 城防值 / 200
```

- 城墙设施等级、设施耐久、太守和守城武将继续追加局部修正。
- 攻城器械对结构使用 1.35 系数。
- 普通野战兵种对结构只使用自身对城系数。
- 没有攻城器械时，单个城防阶段对城墙或城门造成的耐久损失上限为 8%。
- 步兵保护不足时，攻城器械受到守军突击和弓箭的损失明显增加。

### 12.5 攻方命令

攻方至少可以选择：

- 强攻城墙。
- 集中攻门。
- 云梯多点攀城。
- 掘地坏墙。
- 火攻城门。
- 围困消耗。
- 围点打援。
- 主动撤围。

### 12.6 守方命令

守方至少可以选择：

- 坚守墙头。
- 增援城门。
- 保存预备队。
- 出城突袭攻城器械。
- 夜间突围。
- 退守内城。
- 等待援军。

守方实时判断当前有效兵力：守军达到攻军的 `1.6` 倍时，步兵、枪兵和骑兵可开城主动迎战，弓兵继续留在墙头提供射击；出城部队不享受城防减伤。若优势缩小到 `1.1` 倍或以下，出城部队自动退回城内，同一场战斗不反复出城刷状态。出城和退守必须写入实时事件与战报说明。

玩家防守时同样进入战前军议，不得只允许玩家作为攻方操作。

### 12.7 倒计时决胜

- 正式战斗使用 60 秒倒计时和双方总兵力条，当一方兵力条耗尽时可以提前结束。
- 攻方必须先将外墙或城门至少一项降到 0，才能进入城内并取得攻城胜利。
- 倒计时结束时，攻方只有在已经破城且剩余兵力比例高于守方时胜利；城防未破或守方比例不低于攻方时，由守方获胜。
- 本轮产生明确胜负，不再把未破城结果自动转为下一月重复播放的完整攻城战。

### 12.8 城池类型与背景

攻城背景根据城池地图图标的地域家族选择：

| 地域家族 | 城堡背景 |
| --- | --- |
| 中原、华北、东北 | 中原夯土砖城 |
| 西凉、雍州 | 西凉边塞城 |
| 江南、荆州、益州、交州 | 江南/南方城池 |
| 南中、云南 | 南蛮山寨城 |

- 城池位于守方一侧。
- 玩家进攻时，敌城位于左侧；玩家防守时，己城位于右侧并镜像布局。
- 背景中的外墙、城门和内城需要与正式三层状态叠加，不只作为静态图片。

### 12.9 攻城结果

战报需要记录：

- 各层开始和结束耐久。
- 形成缺口的方式和负责武将。
- 双方每个阶段的兵种损失。
- 是否继续围城。
- 城池设施损伤。
- 守城武将撤退、受伤或被俘。
- 缴获、民心、治安和占领后整合影响。

## 13. 动画与战斗画面

### 13.1 总体布局

- 敌方固定在左，玩家固定在右。
- 两侧展示主将大肖像、副将小肖像、职责、士气、兵力和对位状态。
- 中间负责阵型布置、战术选择和正式战斗演出。
- 野战军团向中线推进；攻城军向守方城墙或城门推进。
- 中间战场使用连续大战场而非单屏布景：野战逻辑空间映射到约 `2400×1350` 的战场画布，默认视野只覆盖其中一部分，玩家通过镜头移动观察不同战线。
- 战场支持滚轮缩放、中键拖动画面、点击小地图跳转和镜头归中；顶部兵力、倒计时和城防信息固定在屏幕空间，不随战场移动。
- 右下角小地图始终显示双方战斗队、当前选择和镜头范围，缩到战略总览时能够判断 1–40 个战斗队的前中后排和左右翼关系，放大后再进行局部指挥。
- 所有点选、框选、右键移动、集火、箭矢和伤亡数字必须经过同一镜头坐标变换，镜头缩放不得改变领域层位置、射程、速度或结算结果。
- 正式开战后收起战前阵型配置和武将卡，指挥栏固定排列在战场上方，避免页面滚动把小地图或实时军令区挤出视野；战后贡献仍在战报中完整展示。
- 攻城画面必须明确绘制城墙分界和城内守备区；通常守军在城内列阵，触发兵力优势出城规则的步、枪、骑军可以越过城门迎战，弓兵仍留城守墙；城防条显示在守城方一侧。
- 交战开始后启用轻量 RTS 指挥：左键点选、拖框多选、`Shift` 追加，右键敌军集火、右键地面移动；指挥栏提供全选、攻击目标、自由进攻和原地固守。
- 实时军令必须写入领域层并持续影响后续固定步长模拟，禁止只播放一段与结算无关的命令动画。
- 攻城未破时，攻方移动目标不得越过外墙/城门接敌线；未获得出城状态的守军不得离开城内守备区，已出城部队允许在城外接受 RTS 移动和集火命令。

### 13.2 武将展示

- 所有参战武将均以独立骑将单位进入战场，不生成步行武将，也不把武将合并进普通步兵或骑兵图标。
- 普通武将根据有效属性选择通用猛将、统帅或军师骑乘图集；刘备、关羽、张飞、吕布等历史名将优先使用专属骑乘模型与八帧动画。
- 骑将跟随其负责战斗队移动；近战武将纵马突击，军师在马上挥令并组织远程压制。未破城且双方没有出城接战时，骑将不得隔墙直接造成伤害。
- 骑将单次战力由有效统率、武力和智力计算：猛将更依赖武力，军师更依赖智力；历史名将获得有限的专属修正。骑将明显强于单个普通士兵，但不能独自替代数百人的战斗队。
- 主将使用大肖像，显示主将倍率、统兵容量和全军职责。
- 两名副将必须各自显示肖像、职责名称、负责兵种/阵位和预计贡献。
- 副将触发作用时，其肖像和对应战斗队同时高亮，并显示简短命令条，例如“张飞率前军突破”“诸葛亮识破火攻”。
- 点击任意武将可以查看该武将当前贡献明细，不暂停结算。
- 战后所有出战武将都必须出现在贡献列表中。

### 13.3 五兵种动画

现有每套 8 帧图集统一解释为：

- 第 1–4 帧：行进循环。
- 第 5–8 帧：兵种动作循环。

| 兵种 | 推进动作 | 接战动作 | 特殊表现 |
| --- | --- | --- | --- |
| 步兵 | 持盾行进 | 格挡、盾击、挥砍 | 盾阵时队列紧密 |
| 枪兵 | 持枪推进 | 架枪、突刺 | 克制骑兵时显示拒马式阵线 |
| 弓兵 | 在后排跟随前线移动 | 进入射程后停驻、搭箭、拉弓、齐射 | 箭矢飞行与落点独立表现；目标出射程才继续移动 |
| 骑兵 | 骑马行进 | 加速冲锋 | 侧击时走弧线，不与步兵完全重叠 |
| 攻城器械 | 士兵推行 | 冲车撞击 | 只在攻城或营寨战进入完整攻击帧 |

### 13.4 大量战斗队与群阵表现

- 正式实例数量由 §5.3 的人数公式决定，每侧 1–40 个战斗队，不再设置会抹平人数差距的最低 6 队。
- 每个战斗队由旗帜、状态标识和 3–6 名表现士兵组成；表现士兵共享战斗队的逻辑状态，但动画起始帧、步幅和站位有轻微差异。
- 同兵种不同战斗队的动画循环错开 0.08–0.25 秒，避免整片士兵像同一张贴图同步闪动。
- 战斗队按纵深排序：后方队伍先绘制、前方队伍后绘制；骑兵侧击沿预设弧线移动，箭矢跨越中景飞行。
- 队形由可复用站位模板生成，例如双列、三列、横阵、纵队、楔形、散开，不为每一名士兵计算碰撞。
- 队伍减员达到 25%、50%、75% 时逐级减少表现士兵、改变旗帜状态并扩大队列空缺，让玩家不看数字也能感知损耗。
- 战线发生缺口时，相邻战斗队按时间线向内收拢；收拢是阵位切换动画，并会影响侧翼暴露与后排安全。
- 镜头可在关键冲锋、齐射和破门事件中轻微推近，但事件结束后必须返回能看清全局阵型的视野。

### 13.5 画面层次

战场至少包含：

1. 远景背景。
2. 中景城池、道路、河流或山体。
3. 战斗队与队形。
4. 箭矢、尘土、撞击、火焰等效果层。
5. 近景旗帜、烟雾、草木或城墙遮挡层。
6. HUD 和武将肖像层。

攻城时允许部分军团被城墙近景遮挡，以形成前后层次，但正式兵力和状态不得被遮住。

### 13.6 实时演算与表现同步

- 领域层以 0.1 秒固定步长更新坐标、目标、射程、攻击冷却、伤亡和城防；表现层直接读取当前状态，并消费实时产生的 `BattleTimelineEvent`。
- 每次攻击事件包含发生时间、参与战斗队、逻辑位置、目标、武将、动作、数值变化和文本。
- 撞击帧出现时同步显示该事件造成的损耗或结构伤害。
- 战斗队人数降为 0 或士气崩溃时，播放溃散/退出战场，而不是继续战斗。
- 顶部持续显示双方总兵力条和中央倒计时；最终结果面板在实时演算得出胜负后自动出现，或由玩家主动跳过后快速完成同一套固定步长演算。

### 13.7 地形背景

- 道路地形为 `plain` 时使用平原背景。
- `hill` 使用丘陵背景。
- `river` 使用水路/渡口背景。
- `mountain` 使用山地背景。
- 不能继续在所有战报和战斗中硬编码平原背景。

### 13.8 可访问性与性能

- 降低动态效果时关闭大幅镜头位移、屏幕震动和高频闪烁。
- 损耗、阶段和计策必须同时提供文字状态，不只依赖颜色和动画。
- 基准分辨率为 1440×900。
- 每侧最多 40 个正式战斗队、约 240 名表现士兵；优先使用共享图集、对象池、批量绘制和视野外停更控制性能。
- 性能不足时依次降低粒子数量、单队表现士兵和远景动画频率，不得减少正式战斗队数量或改变阵位关系。
- 同一时间最多重点播放 3 组高成本特效，其余交战使用轻量尘土、闪光和数字反馈。
- 战斗播放过程中不得修改存档随机状态两次或因帧率差异改变结果。

## 14. 战前界面

### 14.1 必须展示

- 双方已知兵力与情报区间。
- 我方统兵容量和超限校验。
- 双方已知兵种构成与克制倍率。
- 主将和所有副将的职责贡献。
- 地形倍率。
- 阵型效果和不满足条件的折损。
- 3×5 布阵盘，以及各兵种的前军、中军、后军和左右翼部署。
- 按当前兵力实际展开的战斗队数量、各队代表兵力和阵位容量。
- 选中兵种时显示有效射程带、预期停止位置、掩护关系和可能暴露的战线。
- 各兵种专属队形、自动行动规则、攻击优先级和前排溃散后的处置军令。
- 主战术、兵种条件状态、执行武将和预计触发阶段。
- 训练、士气、疲劳、补给天数。
- 攻城时的城防、城墙设施、城门和预计结构伤害。
- 主要优势、主要风险和胜率区间。

### 14.2 情报限制

- 未侦察到的敌方副将、战术和准确编制不得提前公开。
- 情报不足时显示兵力区间、兵种倾向和“存在未知武将”等提示。
- 正式接战后可逐步揭示敌军信息；战后报告完整公开实际结算因素。

## 15. 战后报告

战后报告至少包含：

1. 战斗类型、地点、地形、回合和随机种子标识。
2. 双方战前、战后总兵力。
3. 分兵种战前、战损、伤兵、溃兵、被俘和幸存人数。
4. 双方主将和所有副将的职责贡献。
5. 每个阶段的战力比、关键事件和士气变化。
6. 阵型、主战术、兵种条件和执行结果。
7. 兵种克制、地形、补给和疲劳的具体倍率。
8. 攻城三层状态和城市是否易主。
9. 武将受伤、被俘、立功、救援和关系变化。
10. 影响胜负的前三项优势和前三项风险。

武将贡献示例：

```text
赵云 · 主将
全军指挥 +10.8% · 阵型完成度 +7% · 溃兵收拢 184 人

张飞 · 前锋
负责步兵前军 · 首次冲击 +6.5% · 前军接战减损 2.8%

诸葛亮 · 军师
火攻效果 +11% · 识破敌方佯退 · 攻城结构伤害 +4%
```

## 16. AI 规则

- AI 使用与玩家完全相同的统兵容量、武将职责、兵种克制、战术和攻城层级。
- AI 出征时从真实同城可用武将中选择主将和副将，并将其状态设为出征中。
- AI 根据武将属性、特性、兵种和目标选择职责，不固定把第二高统率武将当作无差别加成。
- AI 必须生成合法的 3×5 布阵方案，保证远程和器械有前排掩护，并根据敌方骑兵数量决定枪兵正面或翼侧拒止。
- AI 与玩家使用相同的射程、停驻、换线、冲锋和溃退状态机，不得让 AI 弓兵无视射程或友军遮挡直接输出。
- AI 必须考虑留守武将；不能把一座城所有高统率武将全部带走后仍获得完整守城加成。
- AI 选择攻城策略时考虑攻城器械占比、粮草、城防、援军时间和持续围城成本。
- AI 不得读取玩家尚未公开的战术和职责分配。

## 17. 数据与存档要求

### 17.1 ArmyData

需要补充或正式启用：

- `Composition`：分兵种现役人数。
- `WoundedByTroop`：分兵种伤兵。
- `RoutedByTroop`：分兵种溃兵。
- `Training`、`Morale`、`Fatigue`。
- `FormationId`。
- `OfficerAssignments`：武将、职责、负责兵种或阵位。
- `SupplyDays` 或可推导的每日粮耗。
- `SiegeStateId`。

### 17.2 PendingBattle / BattleContext

至少保存：

- 战斗 ID、类型、地点、道路、地形。
- 双方军团或守城方快照。
- 玩家是否需要下令。
- 当前战前选择。
- 战斗随机状态。
- 是否已经生成时间线。

### 17.3 BattleTimelineEvent

至少保存：

- 阶段。
- 时间点与持续时间。
- 发起方、战斗队、目标战斗队和武将。
- 发起方和目标的逻辑起点、终点、战线和纵深。
- 动画动作。
- 兵力、士气、结构和状态变化。
- 展示文本和音效标识。

### 17.4 FormationPlan / BattleUnitGroup

`FormationPlan` 至少保存：

- 全军阵型 ID、3×5 区域占用和阵位容量。
- 每个兵种的部署区域、专属队形、攻击优先级和进退军令。
- 自动展开前的玩家选择，以及展开后的战斗队列表。
- 阵型完成度、过密、缺口、掩护和侧翼暴露状态。

`BattleUnitGroup` 至少保存：

- 唯一 ID、所属战略军团、兵种和代表兵力。
- 当前兵力、士气、疲劳和阵型完整度。
- 战线、纵深、逻辑坐标、队形锚点和朝向。
- 最小/最佳/最大射程、攻击间隔和移动速度。
- 当前状态、当前目标、上一个目标和状态开始时间。
- 负责武将、受到的职责修正和关键战斗事件。

### 17.5 BattleReportData

在当前字段基础上补充：

- 双方主副将与职责贡献。
- 双方开战和结束编制。
- 地形、阵型、战术和倍率。
- 分阶段事件摘要。
- 分兵种伤亡。
- 伤兵、溃兵、被俘。
- 攻城层级变化。
- 城防和设施损伤。
- 关系、功勋和健康变化。

### 17.6 存档迁移

- 当前存档结构为 schema v1，战斗系统实现时升级 schema。
- 旧军团没有混编、士气或职责字段时，按其唯一兵种、标准士气、标准阵型和现有主副将生成兼容值。
- 旧战报保持可读，不伪造不存在的阶段事件；界面标记为“旧版简报”。

## 18. 数值约束与平衡目标

- 武将总影响重要但不能取代兵力：全部武将因素对全军有效战力的综合影响建议限制在 0.85–1.25。
- 单名副将对全军最终战力的等效影响通常为 2%–8%。
- 单个特性通常为 2%–6%，同阶段特性合计不超过 15%。
- 单个战术的最终影响限制在 −12% 至 +15%。
- 兵种对位限制在 0.82–1.18。
- 阶段随机波动限制在 0.96–1.04。
- 同等其他条件下，兵力达到对方 2 倍的一方应拥有明显优势；除非遭遇极端地形、断粮、被包围或严重克制，不应被普通随机值翻盘。
- 以同等 5,000 人、同将领和同状态测试时，骑兵对步兵的对位倍率应至少达到 1.08，步兵对骑兵保持 1.00。
- 没有攻城器械且兵力不超过守军 1.2 倍时，不应在一次强攻中稳定攻下高城防城市。

## 19. 验收标准

### 19.1 军团与武将

- 1 名主将和 2 名副将出征时，三人都在战前、时间线和战报中拥有独立贡献。
- 移除任意一名副将后，至少一个对应阶段数值和战后贡献发生确定变化。
- 更换副将职责后，影响转移到对应兵种或阶段，而不是继续作为相同全军加成。
- 守城主将、太守和守城副将均能实际影响守城。
- 武将不能同时在两个军团、城内职责和战斗中出现。

### 19.2 兵种与公式

- 五兵种均进入基础系数、地形、克制、动画和分兵种损耗。
- 固定测试中枪兵强克骑兵、骑兵克步兵和弓兵、弓兵克无掩护步枪，步兵不克骑兵。
- 只有进入有效射程且拥有合法目标的战斗队能造成伤害；调整初始距离后，第一轮攻击时间随之确定变化。
- 相同军团采用不同布阵时，接敌顺序、兵种存活、阶段损耗和战果必须产生可解释的变化。
- 同等条件下，弓兵置于完整步枪前排之后的损失，应明显低于弓兵置于前军或无掩护翼侧的损失。
- 枪兵正面密集枪阵可显著降低骑兵冲锋收益，但骑兵完成侧后迂回时枪兵克制效果下降。
- 相同输入和随机状态产生完全相同的阶段损耗、武将事件和最终结果。
- 战前预估使用正式计算器，不在界面另写一套近似常量。

### 19.3 动画

- 四类道路地形分别加载正确背景。
- 五兵种分别播放行进和攻击动作。
- 6,000、12,000、24,000 人军团应分别展开约 10、20、40 个战斗队，且同兵种能形成多排、多列或两翼分队。
- 双方战斗队按阶段和阵位向交战区域移动，损耗数字与正式时间线一致。
- 弓兵默认位于后排，在目标进入有效射程后停步射击；目标离开极限射程后才继续移动或换目标。
- 前排未溃散时，后排弓兵不得越过前排走入近战；前排崩溃后按已选军令后撤、固守或向中军靠拢。
- 箭矢落点、骑兵冲击帧、步枪接敌帧和攻城撞击帧必须与对应正式损耗事件同步。
- 大量战斗队不能全部叠在中线同一点，前中后排和左右翼在全局镜头中持续可辨认。
- 所有出战武将都展示肖像，职责触发时有明确视觉反馈。
- `×1`、`×2`、跳过和降低动态效果不改变最终结果。

### 19.4 攻城

- 城池按地域加载正确城堡背景。
- 外城墙、城门、内城状态可以独立变化并保存到下一月。
- 未形成有效缺口时不能直接占领城池。
- 攻城器械、攻城督、步兵护卫和守城武将均有独立作用。
- 玩家作为守方时也能进入布阵和守城命令界面。

### 19.5 战报与存档

- 战报能解释至少三项胜因、三项风险和每名武将的贡献。
- 战斗进行到战前军议、时间线生成后、持续围城中三个状态均可保存和恢复。
- 旧存档可迁移，旧战报仍可查看。
- 自动演进与玩家战斗使用相同领域计算器。

## 20. 建议实施顺序

### 第一阶段：领域规则

1. 拆分独立战斗计算器，不再把全部逻辑放在 `GameRuntime.ResolveBattle`。
2. 增加军团训练、士气、疲劳、分兵种伤亡和武将职责。
3. 完成兵种、地形、阵型、战术和全武将贡献公式。
4. 增加确定性阶段时间线和详细战报。

### 第二阶段：接战暂停与战斗画面

1. 增加 `PendingBattle`，玩家战斗先暂停再下令。
2. 完成 3×5 布阵盘、兵种部署区域、专属队形和自动展开预览。
3. 增加逻辑坐标、射程、目标选择及战斗队状态机，不使用物理碰撞。
4. 完成敌左我右、全武将肖像、布阵和战术界面。
5. 接入五兵种图集、四地形背景、大量战斗队与实时损耗。

### 第三阶段：分层攻城

1. 增加外墙、城门、内城和持续围城状态。
2. 接入四类地域城堡与镜像守城布局。
3. 完成攻守命令、援军、撤围和城池设施损伤。

### 第四阶段：平衡与 AI

1. AI 使用武将职责和混合编制。
2. 建立固定战斗矩阵和批量种子测试。
3. 调整人数、克制、武将、战术和攻城节奏。
4. 完成旧存档迁移和长局压力测试。

## 21. 已确定决策

- 正式版本以 Godot/C# 当前工程为实现目标。
- 优化首版保持 1 主将＋最多 2 副将，但所有出战武将必须影响战局。
- 战斗采用“自动接敌＋战斗队级 RTS 指挥”；不操作时可完整自动演算，操作时玩家能实时改变移动、目标和接敌方式。
- 一支战略军团按每约 600 人一个战斗队展开，每侧正式显示 1–40 个战斗队，小股部队不会被强制扩成 6 队。
- 玩家战前布置兵种区域和队形，开战后可点选/框选战斗队并下达移动、集火、自由进攻和原地固守命令。
- 战场采用 3 个纵深、5 条战线和逻辑距离，不做逐兵碰撞与寻路。
- 弓兵默认置于后排，目标进入射程后停止推进并射击，目标离开射程后才重新移动。
- 玩家作为攻方或守方都可以进入战斗画面。
- 步兵不克制骑兵；枪兵强克骑兵，骑兵克制步兵和弓兵，并擅长追击溃军。
- 攻城必须经过分层城防，不允许只比较一次总分直接易主。
- 领域层按固定时间片实时计算接敌、射程、围杀、伤亡和破城；动画直接呈现当前领域状态和实时事件。
- 天下地图允许选中敌方战略军团并发起一次性拦截；该命令绑定具体军团而非城池，进入无城防加成的实时野战，战后双方收兵回营且不触发城池易主。
