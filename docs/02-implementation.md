# Godot 实现设计

## 1. 技术栈

- Godot 4.7 .NET：场景、输入、渲染、音频和桌面导出。
- C# / .NET 8：游戏规则、运行时状态、存档和界面逻辑。
- JSON：剧本、势力、城池、武将、道路、关隘和事件数据。
- Python + Pillow：离线图片裁切、压缩、图集与预览生成。

项目不再保留 React、Vite、Tauri 或浏览器运行入口。

## 2. 目录与职责

```text
godot/
├── project.godot
├── ThreeKingdomsSimulator.Godot.csproj
├── scenes/                 # Godot 场景
├── scripts/                # C# 代码
├── data/                   # 可直接读取的剧本 JSON
└── assets/
    ├── runtime/            # 运行时优化资源
    ├── generated/          # 生成图片源文件和审阅图
    ├── audio/              # 音乐
    └── app-icons/          # 应用图标
tools/                      # Godot 启动与资源处理工具
```

## 3. 运行时结构

- `Main` 创建长期存在的页面和全局音频，不在页面切换时反复加载资源。
- `GameRuntime` 持有游戏会话、执行玩家命令、推进月份、AI、事件和胜负判断。
- `ScenarioLoader` 从 `res://data/first-rivalry.json` 载入唯一剧本数据源。
- `WorldMapView` 使用原生 `Node2D` 绘制地图、道路、城池、关隘和军团，拖动与缩放只改变地图根节点变换。
- `CityManagementView` 与 `FeatureScreens` 负责原生 Godot UI。
- `SaveService` 负责自动档、手动档和完整 `GameSession` 序列化。
- `RuntimeSelfTest` 覆盖关键命令、月份推进、事件和存读档闭环。

天下地图加载低饱和淡墨黑白 `world-map-bg-v4.png`，城池与关隘采用逆缩放标记并设置总览最小尺寸，保证缩到天下全图时仍能辨认。敌城摘要只显示“出征此城”，通过 `ExpeditionRequested` 将目标城 ID 带入出征页；己方城仍进入内政。

## 4. 数据与资源原则

- `godot/data/first-rivalry.json` 是运行时剧本的唯一来源，不再从旧 TypeScript 导出。
- 代码只引用 `res://` 路径，不读取仓库外部文件。
- `assets/generated/` 保留原始生成文件；游戏优先加载 `assets/runtime/` 中经过裁切与压缩的资源。
- 武将肖像、城池标记和关隘标记通过 `AssetPaths` 统一生成路径。
- 新资源处理脚本必须把输出写入 `godot/assets/`。

## 5. 验证门槛

每次修改至少按风险运行以下命令：

```bash
./tools/run-godot.sh check
./tools/run-godot.sh smoke

```

涉及 UI 或素材时，还应在编辑器或正式窗口中检查 1440×900 基准布局、缩放、资源加载和交互反馈。

## 6. 势力共用府库与城池内政实现

- `ScenarioModels.cs` 的 `CityData` 保存城池发展、驻军、设施、城务额度、太守治理、定位、当月武将行动和城市台账；其中剧本金粮只用于汇总开局势力府库，进入游戏后不再作为独立城市府库流转。
- `DomesticSystem.cs` 负责城务、设施、城池月度产出、状态、太守委任和 AI 内政评分；玩家与 AI 共用同一命令执行器。
- 194 年剧本将人口解释为城池可直接支配的登记人口；下调人口展示值时同步校正税粮换算系数，并降低单次征兵量、目标守军和 AI 出征阈值，避免只改初始数据导致经济或战争停摆。
- 金、粮为势力通用资源。普通城务、设施建设、设施维护、驻军维护、出征军粮、撤兵回粮和官员俸禄均直接结算到所属势力府库；不再有中央与城池间调拨。
- 商业税收按人口与商业值产生每城数百金的月度贡献，与 800～1,600 金的建筑成本处于同一量级；自测要求默认开局首月金收入扣除维护与俸禄后仍至少净增 300 金。
- 粮食自然产出使用 `人口 × 农业 ÷ 27,000`，灌溉渠与粮仓每级追加 120；标准刘备开局自然粮产承担超过一半驻军维护，每座初始城市各有一座一级粮食设施后，总产粮与维护差距控制在 10% 内。
- AI 每月先评估粮食覆盖、财政、治安、民心、疲敝、相邻敌军压力、守军缺口、设施和可用武将，再决定城市方针、定位、建设和城务命令，不再免费增加农业、商业或驻军。
- 太守按职位持续任职，不与武将的驻城、调动或出征状态绑定；太守出征或调动后仍保留原城职位和治理额度加成，但本人不可在此期间执行城务，攻下新城也不会覆盖其原有太守职位。
- `CityManagementView.cs` 提供势力内政总览，以及单城的“总览 / 建筑 / 治理”页签。顶部导航保持在城池详情层之上，“返回天下 / 上一城 / 下一城”均可直接操作；城池状态与台账放在定宽可滚动区域内，不参与页面横向撑宽。建筑页使用可点选的城内地块，显示设施功效、成本、工期，并通过“建造 → 选择武将 → 确认开工”完成委任；开工后设施立即占用所选地块并显示“建造中”和剩余月份，完工后原地转为正式设施。城池等级为 Lv.1～Lv.8，每次升级提高农业、商业、城防与文化，并在 Lv.2/4/6/8 将建造位置从 4 个逐步扩展至 8 个；主动取消升级会全额返还金粮，但不返还本月城务额度。
- 治理页保留治理方式、方针和城市定位；未接入实际规则的“允许邻城支援”开关及其数据字段已移除，城市保底金粮和盈余上缴也已移除。
- `RuntimeSelfTest.cs` 覆盖两城城务隔离、势力府库扣费、初始自然产粮与粮食设施平衡、共用资源下的 AI 决策、俸禄、出征/撤兵/攻城胜负回粮与 60 回合压力演进。

通用功能页使用锚点式滚动容器和可换行 `HFlowContainer`，出征、人才、外交、军师与存档页不再依赖 1350 px 固定内容宽度；内政页的城务、建筑与治理操作行同样允许自动换行，底部导航按可用宽度伸缩。

## 7. 新游戏与胜负规则实现

- `NewGameSetupView.cs` 是正式新游戏入口，列出剧本内全部 16 个势力，并提供标准、宽松、艰难三档难度与每月、每三个月、每年、关闭四种自动存档设置。
- `GameSession.Create` 接收 `NewGameOptions`。玩家中央府库按所选势力全部初始城池的金粮库存汇总；新游戏页同步展示预计首月金粮变化。原剧本默认玩家改为 AI 后，其府库会回到 `FactionTreasuries`，外交列表也会排除当前玩家而重新生成。
- 难度的首版公开效果只调整玩家初始金、粮：宽松为 `+20%`，标准不变，艰难为 `-15%`；威望、城市数据和 AI 规则不做隐藏修改。
- 自动存档依据会话内 `AutoSaveFrequency` 在月末执行；旧档没有该字段时默认每月存档。
- 胜负结算支持统一全部城池即时胜利、零城即时失败，以及控制至少 9 城并在连续 3 次月末结算中保持条件后触发“九城归心”。地图顶栏持续显示当前城数和连续维持月数。
- 当前存档结构使用 schema v9，包含难度、自动存档、九城归心进度、逐城方针委任与城池等级，移除全局玩家托管与全 AI 演进状态；当前阶段不提供旧档迁移或兼容分支。
- `RuntimeSelfTest.cs` 覆盖 16 势力可选性、改选势力后的府库和外交归属、难度资源、自动存档频率、旧档默认值、九城连续三月胜利与中断清零。

## 8. 实时战斗决策实现

- `ExpeditionView` 仅保留五类兵种人数作为兵力输入，军团总兵力实时由阵型合计派生；“全城出击”会把当前城全部驻军填入编制。`GameRuntime.CreateExpedition` 允许玩家清空驻军，统兵容量改为超额士气惩罚与风险提示，AI 的最低留守规则保持不变。
- 天下地图军团标记使用骑兵八帧序列动画；每次行军同时播放角色动作与地图位移，并根据实际移动方向水平翻转，不再显示为一张静止图片。城池区域默认使用更高区域缩放，天下全图中的城池标记与点击框同步缩小，减少邻近城池误点。
- 玩家可以直接点击天下地图上的独立军团，查看兵力、军粮、士气与行程。出征当回合锁定操作；进入下一回合后，每支军团可独立选择继续前进、继续攻城或撤回任意己方城市。主动撤兵、停战召回以及攻城胜负会把剩余军粮一次性归还势力府库并清零军团携粮；野战胜军和正在沿路败退的军团仍为活动军团，继续携带并消耗军粮，败军抵达己城后才回收余粮。
- 点击敌方军团后可选择“出击拦截这支军团”，再到出征页单独配置出发城、武将、兵种与军粮。拦截命令以 `TargetArmyId` 绑定具体目标；抵达后创建 `field` 军团野战，直接使用目标军团的真实编制、兵力、训练、士气、疲劳、补给和随军武将，不把目标城市守军代入战斗。
- `BattleSystem.cs` 使用 0.1 秒固定步长实时推进战斗。战斗队根据逻辑坐标寻找目标，只有进入射程后才会攻击；伤亡、箭矢、攻墙和破门事件在发生时写入时间线，不再预先生成固定动画结果。
- 领域层继续保持 0.1 秒确定性步长，渲染层通过 `PreviousX/Y` 与当前坐标按剩余累积时间进行平滑插值，因此不会因数值步长而每 100ms 跳位；战斗画面把活动时间线改为每帧单次扫描，避免军团数量和事件积累导致后半程卡顿。
- 弓兵齐射使用事件记录的固定发射点与命中点绘制，不再追随移动中的单位漂移。单次齐射按部队规模显示 5–11 支错峰箭矢，采用二次贝塞尔抛物线、方向箭头、阴影拖尾与落点火花，并把伤害浮字延后到箭矢接近命中时显示。
- 姿态、全军阵型、兵种军令、克制、训练、士气、疲劳、补给和武将职责共同进入每次实时攻防；当前版本全部战斗统一按平地演算，不再叠加道路地形或局部树林、坡顶、浅滩倍率。防守姿态、盾墙/坚营战术、盾阵和兵种军令会直接降低承伤，不再只影响进攻端。
- 近战队、弓兵和攻城器械分别限制同时接战数；弓兵进入最小射程内会后撤拉开距离，不能贴身无惩罚齐射。
- 每个战斗队独立保存 0–100 士气；伤亡、包围和骑兵冲击会造成士气震荡，低于 25 后撤、低于 10 溃散，高统率与魅力武将可尝试收拢溃兵。士气与溃散数写入阶段结果和战报。
- 每名上阵武将拥有独立的战场体力、士气和防御：武力与统率提高体力和防御，统率与魅力提高初始士气；武将交锋、阵前反击和所属战斗队受创会分别扣减这些数值。武将体力远高于普通伤害量，体力耗尽或士气低于 20 时退出战线，战场下方同步绘制体力条和士气条。战略层 `Health` 当前固定保持 100，战后不扣减。
- 出征页不再选择姿态或战术；军团接战后统一在战前军议选择姿态、阵型、兵种军令和一个主战术。兵种条件不足时禁止开战，战中不自动切换其他战术。
- 战斗以 60 秒倒计时和双方总兵力条结算。攻方必须先攻破外墙或城门；时间结束时，破城且剩余兵力比例占优才可夺城，否则守方获胜，不再自动生成重复围城战。
- 攻陷城池后驻军只等于攻方剩余现役兵；守方幸存兵不再直接并入征服者驻军。双方军团离开战场时保留按战斗队加权的剩余士气。
- `BattlefieldCanvas` 在攻城背景上额外绘制独立的前景城墙、城门、城楼、裂纹和瓦砾；墙体与城门随实时耐久切换完整、破损和缺口状态，不再只显示一条抽象分界。优势守军可按兵力比主动出城，玩家守城时可下达增援城门、退守内城、出城突袭和保留预备队军令。
- 战前阵型栏只显示玩家军团实际拥有的兵种；切换全军阵型或任一兵种队形时立即重新计算战斗队的战线、纵深和坐标，并在大战场中实时预览。
- 战前军议不再额外显示整块兵力、训练、疲劳、地形和风险基础信息框；战场 HUD 与武将卡保留战斗所需信息，情报不足时仍隐藏敌将身份。
- 武将卡在战前布阵和实时战斗期间始终显示，其职责/特性文字支持鼠标悬浮说明：提示框展示主将、前锋或军师职责的当前战力贡献，并列出该武将个人特性的具体功效、数值与生效条件；未侦明的敌将不生成提示，避免泄露情报。战报汇总时，同一武将的同一特性即使作用于多个战斗阶段也只展示一次。
- `BattlefieldCanvas` 使用高质量 2.5D 大战场：当前野战统一使用开阔平地原画，攻城仍按地域绑定正式原画；领域层 0–1000 坐标只投影到画面下方的可行军地表，单位按纵深缩放，并叠加软阴影、移动扬尘、战旗和兵力反馈。领域坐标映射为“敌军在左、我军在右”，并按玩家攻守方自动翻转；横向图集由阵营独立镜像，动作行按移动/攻击状态选择。普通滚轮用于页面滚动，`Ctrl+滚轮` 或战场缩放按钮才调整镜头；中键拖动、迷你地图、框选和右键军令仍反算到同一领域坐标。
- 战斗与战报使用独立界面：`BattleView` 只承载战前布阵、战场画面和实时军令，不再在画面下方滚动显示实时事件横条或历史战报；战斗完成后由 `Main` 自动切换到 `BattleReportView`，该界面只展示刚结束的本场战果、阶段结算和武将贡献。
- `BattleView` 将全军阵型、临战军略和兵种阵位压缩到同一战前面板，并缩窄武将卡行；速度与跳过只在开战后出现在实时指挥栏。大战场高度为 560 px，减少整页滚动依赖。双方兵力与倒计时使用三块独立紧凑 HUD；独立 `BattleReportView` 隐藏无内容的通用通知栏，直接从胜负卡开始，并提供醒目的“确认战报，返回天下”按钮。
- 军团野战统一按平地推进，界面不显示城墙、城门和内城；胜负依据双方总兵力条、士气和 50 秒倒计时。实时伤害节奏已提高，溃散或撤退分队在脱离战场前仍会承受附近敌军追击。战略层会把道路 ID 与路段日数偏移写入待结算战斗：胜军结算后停在真实交战点并于下一月继续原路线；仍有兵力的败军以行军日数选择距交战点最近的己方城市，重建返程道路并以 `retreating` 状态逐月移动，抵达后才归还兵力、武将与余粮。战报标记为 `field`，不会改变任何城池归属。
- 战斗队约每 600 人展开一队，范围为 1–40 队；400 人只显示 1 队，3,700 人显示 7 队，兵力规模会直接反映在画面队伍数量上。
- 全局主题不再依赖操作系统字体名称，统一加载 `res://assets/fonts/NotoSansSC-VF.otf`。控件文字和 `BattlefieldCanvas` 的直接绘制文字共用该内置字体，确保 macOS、Windows 与导出包显示同一套完整简体中文字形；字体随附 SIL OFL 1.1 许可证。

## 9. 战略 AI 与全势力外交实现

- `StrategicAiSystem.cs` 每月为各势力生成军事与外交候选，按兵力比、留守兵力与武将、金粮覆盖、路线、地形、前线压力、目标价值、关系、信任和共同威胁评分；攻守比低于安全线的目标会被硬性排除。
- AI 出征在执行前重新校验主将、副将、停战、守军、兵力比和军粮，并真实从出发城扣除驻军、从势力府库扣除军粮；每次评估把前三候选、分数和未选原因写入军师界面可见的审计日志。军团按目标城防、道路地形和主将能力生成至多三类混合编制、留守至少一名本城武将，并设置阵型及主备战术。
- `DomesticSystem.cs` 会在城市进入或离开前线后重新评估守备、粮仓、商埠等定位，建设工程按设施类型选择对应能力的执行武将；在途、围城和待战敌军均进入前线压力。
- `BattleSystem.cs` 只从当前真实驻城且可用的武将中选择守将，并按军事能力决定主将，不再将太守职位等同于守城统帅；AI 守方根据兵力优势和敌方编制选择固守、盾墙、箭雨或合围。失效或已被己方控制的进攻目标会让军团撤回，不会卡在零日行军状态。
- `FactionDiplomacy` 保存全部势力两两关系。AI 与 AI 可以通商、停战，停战会阻止新出征并召回双方在途进攻军团；玩家外交继续复用同一条约底层状态。
- 当前存档直接保存全势力两两外交关系；自测覆盖候选排序、评估无副作用、低粮拒绝、AI 与 AI 停战、条约结算和审计日志，不提供旧档迁移分支。

## 10. 武将成长、特性与官职实现

- `OfficerProgressionSystem.cs` 统一提供 1–20 级阅历门槛、确定性成长方案、永久/有效五维查询、文武官阶、俸禄、特性目录和特殊部队目录；业务公式不再直接读取未加成的基础五维。
- 城务、太守月结、建设完成、延揽、侦察和正式战斗会在任务结算后发放阅历；单人普通阅历按月封顶，升级按个人成长方案增加永久五维，达到个人潜力时转为功勋。
- 文职官阶只强化匹配的政治、智力与魅力任务，武职官阶只强化匹配的统率与武力任务。晋升同时校验等级、功勋、冷却、编制名额和财政安全线，转序有一个月适应期。
- 月末统一从所属势力府库逐人支付俸禄并记录台账；资金不足按比例发放、累积欠俸并影响忠诚，人才页支持补发与降职。
- 攻城战和军团野战完成结算时，只调整实际参战武将：胜方忠诚 +1、败方忠诚 -1，平局不变，结果限制在 0～100 并写入战报摘要。
- 普通、内政和传奇特性通过统一条件表进入城务或战斗阶段公式，同一命令/阶段设置总上限；名将专属特性不使用隐藏的全属性倍率。
- 出征页支持白马义从、陷阵营、西凉铁骑、丹阳兵和虎卫。特殊部队是对应基础兵种的真实子编制，以 500 人为单位编成，并受势力和对应兵种人数限制，随实际战损同比减少；匹配武将特性继续提供战斗加成。出征页不再显示“建议统兵”。
- 人才府拆分为总览、招募、调动、官职四个页签；总览以表格列出全势力武将，在野人才集中到招募页，调动页显示起讫城市、道路日数与剩余路程，官职页以“主公—三系主官—九席属官”的树状朝堂组织任命武将。
- 朝堂职位与原有文武官阶并行：官阶继续提供个人条件属性与基础俸禄；唯一朝堂席位追加月俸，并将任职者的对应能力、普通特性和名将专属特性转化为势力光环。丞相系进入全城钱粮与城务公式，大将军和军师将军系进入战斗阶段、士气、武将体力、训练与军粮维护公式；主官提供完整全势力方向，属官提供较窄的专业方向，特性转化强度为主官的 55%。人才府选择候选后会即时显示该武将上任后的真实加成，不再只显示固定职位属性。每个职位只能任命一人，每名武将只能担任一个职位；职位或武将已占用时必须先卸任，不能直接替换或转任。降职或转序导致资格不足时自动卸任。
- 招募方式由目标状态自动决定：已发现的在野人才可登庸、俘虏可劝降、忠诚低于 60 的在任敌将可策反；关系举荐已移除，策反不提供成功率加成。选择目标或执行者时实时显示预计成功率，但不展示内部公式。
- 武将跨城操作称为调动：下令后进入调动状态，月结推进 30 日，到达目标城后恢复任职；调动途中不能执行城务、出征或升降官阶。只有驻留己方城池且未加入军团的武将可以调动。人才府不再提供独立侦察命令。
- 当前存档结构直接使用 schema v9，包含城池等级、等级成长、官职序列、朝堂职位、特殊部队、武将调动和逐城治理方式，不包含全局玩家托管或全 AI 演进状态；当前阶段不提供旧档迁移或兼容分支。
