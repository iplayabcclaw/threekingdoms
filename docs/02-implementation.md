# Godot 实现设计

## 1. 技术栈

- Godot 4.7 .NET：场景、输入、渲染、音频和桌面导出。
- C# / .NET 8：游戏规则、运行时状态、存档和界面逻辑。
- JSON：剧本、势力、城池、武将、道路、关隘和事件数据。
- Python + Pillow：离线图片裁切、压缩、图集与预览生成。

项目不再保留 React、Vite、Tauri 或浏览器运行入口。

## 2. 目录与职责

```text
godot/
├── project.godot
├── ThreeKingdomsSimulator.Godot.csproj
├── scenes/                 # Godot 场景
├── scripts/                # C# 代码
├── data/                   # 可直接读取的剧本 JSON
└── assets/
    ├── runtime/            # 运行时优化资源
    ├── generated/          # 生成图片源文件和审阅图
    ├── audio/              # 音乐
    └── app-icons/          # 应用图标
tools/                      # Godot 启动与资源处理工具
```

## 3. 运行时结构

- `Main` 创建长期存在的页面和全局音频，不在页面切换时反复加载资源。
- `GameRuntime` 持有游戏会话、执行玩家命令、推进月份、AI、事件和胜负判断。
- `ScenarioLoader` 从 `res://data/first-rivalry.json` 载入唯一剧本数据源。
- `WorldMapView` 使用原生 `Node2D` 绘制地图、道路、城池、关隘和军团，拖动与缩放只改变地图根节点变换。
- `CityManagementView` 与 `FeatureScreens` 负责原生 Godot UI。
- `SaveService` 负责自动档、手动档和完整 `GameSession` 序列化。
- `RuntimeSelfTest` 覆盖关键命令、月份推进、事件和存读档闭环。

天下地图加载低饱和淡墨黑白 `world-map-bg-v4.png`，城池与关隘采用逆缩放标记并设置总览最小尺寸，保证缩到天下全图时仍能辨认。敌城摘要只显示“出征此城”，通过 `ExpeditionRequested` 将目标城 ID 带入出征页；己方城仍进入内政。

## 4. 数据与资源原则

- `godot/data/first-rivalry.json` 是运行时剧本的唯一来源，不再从旧 TypeScript 导出。
- 代码只引用 `res://` 路径，不读取仓库外部文件。
- `assets/generated/` 保留原始生成文件；游戏优先加载 `assets/runtime/` 中经过裁切与压缩的资源。
- 武将肖像、城池标记和关隘标记通过 `AssetPaths` 统一生成路径。
- 新资源处理脚本必须把输出写入 `godot/assets/`。

## 5. 验证门槛

每次修改至少按风险运行以下命令：

```bash
./tools/run-godot.sh check
./tools/run-godot.sh smoke
./tools/run-godot.sh test
```

涉及 UI 或素材时，还应在编辑器或正式窗口中检查 1440×900 基准布局、缩放、资源加载和交互反馈。

## 6. 势力共用府库与城池内政实现

- `ScenarioModels.cs` 的 `CityData` 只保存城池发展、驻军、设施、城务额度、太守治理、定位、当月武将行动和城市台账；城池不再持有独立金粮。
- `DomesticSystem.cs` 负责城务、设施、城池月度产出、状态、太守委任和 AI 内政评分；玩家与 AI 共用同一命令执行器。
- 194 年剧本将人口解释为城池可直接支配的登记人口；下调人口展示值时同步校正税粮换算系数，并降低单次征兵量、目标守军和 AI 出征阈值，避免只改初始数据导致经济或战争停摆。
- 金、粮为势力通用资源。普通城务、设施建设、设施维护、驻军维护、出征军粮、撤兵回粮和官员俸禄均直接结算到所属势力府库；不再有中央与城池间调拨。
- AI 每月先评估粮食覆盖、财政、治安、民心、疲敝、相邻敌军压力、守军缺口、设施和可用武将，再决定城市方针、定位、建设和城务命令，不再免费增加农业、商业或驻军。
- `CityManagementView.cs` 提供势力内政总览，以及单城的“总览 / 建筑 / 治理”页签。建筑页使用可点选的城内地块，显示设施功效、成本、工期，并通过“建造 → 选择武将 → 确认开工”完成委任。
- 治理页保留治理方式、方针、城市定位和邻城支援；城市保底金粮和盈余上缴已移除。
- `RuntimeSelfTest.cs` 覆盖两城城务隔离、势力府库扣费、共用资源下的 AI 决策、俸禄、出征/撤兵粮草与 60 回合压力演进。

通用功能页使用锚点式滚动容器和可换行 `HFlowContainer`，出征、人才、外交、军师与存档页不再依赖 1350 px 固定内容宽度；内政页的城务、建筑与治理操作行同样允许自动换行，底部导航按可用宽度伸缩。

## 7. 新游戏与胜负规则实现

- `NewGameSetupView.cs` 是正式新游戏入口，列出剧本内全部 16 个势力，并提供标准、宽松、艰难三档难度与每月、每三个月、每年、关闭四种自动存档设置。
- `GameSession.Create` 接收 `NewGameOptions`。玩家中央府库从所选势力的初始府库建立；原剧本默认玩家改为 AI 后，其府库会回到 `FactionTreasuries`，外交列表也会排除当前玩家而重新生成。
- 难度的首版公开效果只调整玩家初始金、粮、装备：宽松为 `+20%`，标准不变，艰难为 `-15%`；威望、城市数据和 AI 规则不做隐藏修改。
- 自动存档依据会话内 `AutoSaveFrequency` 在月末执行；旧档没有该字段时默认每月存档。
- 胜负结算支持统一全部城池即时胜利、零城即时失败，以及控制至少 9 城并在连续 3 次月末结算中保持条件后触发“九城归心”。地图顶栏持续显示当前城数和连续维持月数。
- 新游戏与胜负切片曾将存档升级至 schema v4，新增 `Difficulty`、`AutoSaveFrequency` 与 `NineCityControlMonths`；当前总版本为 schema v6，旧档载入仍会补齐标准难度、每月自动档与零进度。
- `RuntimeSelfTest.cs` 覆盖 16 势力可选性、改选势力后的府库和外交归属、难度资源、自动存档频率、旧档默认值、九城连续三月胜利与中断清零。

## 8. 实时战斗决策实现

- `ExpeditionView` 仅保留五类兵种人数作为兵力输入，军团总兵力实时由阵型合计派生；“全城出击”会把当前城全部驻军填入编制。`GameRuntime.CreateExpedition` 允许玩家清空驻军，统兵容量改为超额士气惩罚与风险提示，AI 的最低留守规则保持不变。
- 天下地图军团标记使用骑兵八帧序列动画；每次行军同时播放角色动作与地图位移，并根据实际移动方向水平翻转，不再显示为一张静止图片。
- 玩家可以直接点击天下地图上的独立军团，查看兵力、军粮、士气与行程。出征当回合锁定操作；进入下一回合后，每支军团可独立选择继续前进、继续攻城或撤回任意己方城市，撤兵会回收剩余兵力、军粮和随军武将。
- 点击敌方军团后可选择“出击拦截这支军团”，再到出征页单独配置出发城、武将、兵种与军粮。拦截命令以 `TargetArmyId` 绑定具体目标；抵达后创建 `field` 军团野战，直接使用目标军团的真实编制、兵力、训练、士气、疲劳、补给和随军武将，不把目标城市守军代入战斗。
- `BattleSystem.cs` 使用 0.1 秒固定步长实时推进战斗。战斗队根据逻辑坐标寻找目标，只有进入射程后才会攻击；伤亡、箭矢、攻墙和破门事件在发生时写入时间线，不再预先生成固定动画结果。
- 领域层继续保持 0.1 秒确定性步长，渲染层通过 `PreviousX/Y` 与当前坐标按剩余累积时间进行平滑插值，因此不会因数值步长而每 100ms 跳位；战斗画面把活动时间线改为每帧单次扫描，避免军团数量和事件积累导致后半程卡顿。
- 弓兵齐射使用事件记录的固定发射点与命中点绘制，不再追随移动中的单位漂移。单次齐射按部队规模显示 5–11 支错峰箭矢，采用二次贝塞尔抛物线、方向箭头、阴影拖尾与落点火花，并把伤害浮字延后到箭矢接近命中时显示。
- 姿态、全军阵型、兵种军令、克制、地形、训练、士气、疲劳、补给和武将职责共同进入每次实时攻击；多个友军锁定同一近邻目标时产生有限围杀加成。
- 主战术条件不足、实时损失过高或接战兵力比过低时切换备用战术，切换原因进入正式战报。
- 战斗以 60 秒倒计时和双方总兵力条结算。攻方必须先攻破外墙或城门；时间结束时，破城且剩余兵力比例占优才可夺城，否则守方获胜，不再自动生成重复围城战。
- `BattlefieldCanvas` 固定显示攻城背景和城墙分界；守军部署在城内守备区且不会主动出城。城防条始终显示在守城方一侧，双方总兵力条分列顶部，倒计时位于中央。
- `BattlefieldCanvas` 将 0–1000 领域坐标投影到 `2400×1350` 连续大战场，使用独立镜头完成滚轮缩放、中键拖动、迷你地图跳转和镜头归中；HUD 保持屏幕固定，选框、点选与军令目标会反算回同一领域坐标。
- `BattleView` 在实时交战时收起战前阵型栏和武将卡，把 RTS 指挥栏移动到大战场上方，并在新战斗进入时归零页面滚动，使 1440×900 基准窗口无需滚动即可看到主要战线、镜头 HUD 和小地图。
- 军团野战使用对应地形背景，双方均会向接敌距离推进，界面不显示城墙、城门和内城；胜负只依据双方总兵力条与 60 秒倒计时。战后双方剩余兵力和武将各自回到出发城，战报标记为 `field`，不会改变任何城池归属。
- 战斗队约每 600 人展开一队，范围为 1–40 队；400 人只显示 1 队，3,700 人显示 7 队，兵力规模会直接反映在画面队伍数量上。
- 全局主题不再依赖操作系统字体名称，统一加载 `res://assets/fonts/NotoSansSC-VF.otf`。控件文字和 `BattlefieldCanvas` 的直接绘制文字共用该内置字体，确保 macOS、Windows 与导出包显示同一套完整简体中文字形；字体随附 SIL OFL 1.1 许可证。

## 9. 战略 AI 与全势力外交实现

- `StrategicAiSystem.cs` 每月为各势力生成军事与外交候选，按兵力比、留守兵力、金粮覆盖、路线、地形、前线压力、目标价值、关系、信任和共同威胁评分。
- AI 出征在执行前重新校验主将、停战、守军和军粮，并真实从出发城扣除兵粮；每次评估把前三候选、分数和未选原因写入军师界面可见的审计日志。
- `FactionDiplomacy` 保存全部势力两两关系。AI 与 AI 可以通商、停战，停战会阻止新出征并召回双方在途进攻军团；玩家外交继续复用同一条约底层状态。
- 旧档从原玩家中心外交数据迁移到全势力关系；自测覆盖候选排序、评估无副作用、低粮拒绝、AI 与 AI 停战、条约结算和审计日志。

## 10. 武将成长、特性与官职实现

- `OfficerProgressionSystem.cs` 统一提供 1–20 级阅历门槛、确定性成长方案、永久/有效五维查询、文武官阶、俸禄、特性目录和特殊部队目录；业务公式不再直接读取未加成的基础五维。
- 城务、太守月结、建设完成、延揽、侦察和正式战斗会在任务结算后发放阅历；单人普通阅历按月封顶，升级按个人成长方案增加永久五维，达到个人潜力时转为功勋。
- 文职官阶只强化匹配的政治、智力与魅力任务，武职官阶只强化匹配的统率与武力任务。晋升同时校验等级、功勋、冷却、编制名额和财政安全线，转序有一个月适应期。
- 月末按“驻城城市府库优先、其余中央府库”确定唯一支付方并逐人记录俸禄；资金不足按比例发放、累积欠俸并影响忠诚，人才页支持补发与降职。
- 普通、内政和传奇特性通过统一条件表进入城务或战斗阶段公式，同一命令/阶段设置总上限；名将专属特性不使用隐藏的全属性倍率。
- 出征页支持白马义从、陷阵营、西凉铁骑、丹阳兵和虎卫。特殊部队是对应基础兵种的真实子编制，以 500 人为单位消耗中央军备，并随实际战损同比减少。
- 人才府拆分为总览、招募、调动、官职四个页签；总览以表格列出全势力武将，在野人才集中到招募页，调动页显示起讫城市，官职页以“主公—三系主官—九席属官”的树状朝堂组织任命武将。
- 朝堂职位与原有文武官阶并行：官阶提供基础俸禄，唯一朝堂席位追加月俸与匹配场景的有效属性；改任会自动卸任该席位前任，降职或转序导致资格不足时自动卸任。
- 招募成功率按“基础 10 + 执行者魅力 45% + 君主魅力 15% + 执行者智力 10% + 执行者政治 5% - 目标忠诚 45% + 招募方式 + 共同理念”计算并限制在 5%–95%，招募前不展示预计概率；武将调动明确显示“武将、从原城至目标城”。
- 存档版本升级至 schema v6；旧档会确定性补齐等级、阅历、成长原型/方案、潜力、官职序列、朝堂职位与特殊部队空表，不降低已有基础五维。
