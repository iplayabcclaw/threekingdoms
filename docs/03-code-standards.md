# C# 与 Godot 代码规范

## 1. 基本约定

- 开启可空引用类型，禁止用无意义的 `!` 掩盖可空问题。
- 类型、方法和公开成员使用 `PascalCase`，私有字段使用 `_camelCase`，局部变量使用 `camelCase`。
- 一个文件聚焦一个主要职责；复杂界面拆成可命名的构建方法或独立节点类。
- 领域规则保持确定性。随机结果必须来自会话种子，不得临时创建不可复现的随机源。

## 2. Godot 边界

- 场景负责稳定的节点层级，动态数据和行为由 C# 构建与绑定。
- 高频更新不得反复重建完整 UI 或地图节点；优先更新已有节点的属性或 transform。
- 资源路径集中在 `AssetPaths` 或明确的页面常量中，避免到处拼接字符串。
- 节点信号在创建处连接，回调名称应描述玩家意图。
- 读取项目资源使用 `res://`，存档使用 `user://`。

## 3. 领域与数据

- `ScenarioModels` 描述输入数据，`GameRuntime` 负责状态变化，界面不得绕过运行时直接改会话。
- 所有 ID 引用必须在加载或自测阶段验证；空剧本、缺失城市和无效资源路径应尽早失败。
- JSON 字段调整时同步更新模型、加载逻辑、自测与文档。
- 存档结构变更必须考虑旧版本迁移，不得静默丢弃玩家数据。

## 4. 界面与可用性

- 文本、数值和按钮使用 Godot 原生控件呈现，不能把正式信息烘焙进背景图。
- 颜色不能成为唯一状态提示；搭配文字、图标、边框或形状。
- 交互控件必须有清晰的禁用态、选中态和失败反馈。
- 1440×900 为基准，同时避免硬编码导致窗口缩放后内容被裁切。

## 5. 资源

- 原始生成图放入 `godot/assets/generated/`，优化后的运行时资源放入 `godot/assets/runtime/`。
- 不覆盖既有生成源文件；迭代使用 `-v2`、`-v3` 等版本后缀。
- 新增或替换素材时更新 `docs/05-generated-assets.md`。
- 不提交 `.godot/`、`bin/`、`obj/` 等可再生缓存。

## 6. 验证

- C# 修改：至少运行 `./tools/run-godot.sh check`。
- 规则、存档、事件或回合修改：运行 `./tools/run-godot.sh test`。
- 场景、资源或启动修改：运行 `./tools/run-godot.sh smoke` 并进行窗口检查。
- Python 工具修改：运行 `python3 -m py_compile tools/*.py`，并对实际输出做文件数量和路径复核。
