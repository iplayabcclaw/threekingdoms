# 产品需求文档

## 1. 产品定位

暂定名：**《三国：山河逐鹿》**。

类型为桌面优先的单机势力经营策略游戏。玩家扮演一方君主，通过治理城池、招募武将、调配资源、经营关系和发动战争，逐步控制天下。

核心体验以势力经营和有代价的决策为主，战斗阶段辅以战斗队级轻量 RTS 指挥：玩家既要决定把谁放在什么位置、何时出兵和投入多少粮草，也能在交战中通过移动、集火与固守改变局部战局。

## 2. 设计支柱

### 2.1 人才决定势力上限

武将既是数值单位，也是有性格、关系和诉求的人。高能力武将可能野心大、派系复杂或难以相处；忠诚但平庸的旧部也有稳定后方的价值。

### 2.2 城池是战争的根基

钱粮、人口、治安、民心和补给共同决定扩张能力。连续征战会消耗国力，刚占领的城市不能立刻变成稳定后方。

### 2.3 战斗结果必须可理解

第一版不进入战场。出征前展示双方已知战力、关键优势和风险区间；结算后展示真正影响胜负的因素，不允许只弹出一个无法解释的随机结果。

### 2.4 天下会自行变化

AI 势力拥有发展目标、外交倾向和安全判断，会结盟、背叛、争夺城市和招揽人才。玩家不行动时，世界也会演变。

## 3. 目标用户与体验时长

- 喜欢三国题材、经营模拟和大战略的玩家。
- 单回合目标耗时 2 至 5 分钟。
- 单局第一版目标时长 4 至 8 小时。
- 支持随时保存和退出，不依赖连续长时间操作。

## 4. 第一版内容规模

| 内容 | 目标规模 |
| --- | ---: |
| 城池 | 33 |
| 势力 | 16 |
| 武将 | 112 |
| 基础兵种 | 5 |
| 城市设施 | 10 至 12 |
| 战术 | 12 |
| 动态事件 | 40 以上 |
| 历史剧本 | 1 个虚实结合的小型群雄剧本 |

规模用于验证闭环，不代表最终上限。

## 5. 核心循环

```text
查看天下局势
  → 安排内政与任命
  → 搜索、招募或安抚武将
  → 外交或谋略
  → 组织出征并查看战力预估
  → 战斗评分结算与战报
  → AI 势力行动、事件发生、月份推进
  → 重新评估局势
```

一个回合代表一个月。春耕、秋收、冬季行军和灾害会影响不同月份的收益与风险。

## 6. 开局

第一版提供：

- 选择当前“初平群雄”剧本中的 16 个预设势力之一。
- 查看势力难度、初始城池、君主特性、武将和资源。
- 选择“标准”“宽松”“艰难”三档难度。
- 设置自动存档频率。

自建君主与历史年份剧本放在第二阶段，不阻塞首个可玩版本。

## 7. 天下地图

### 7.1 必要信息

- 城池名称、归属旗色、驻军规模和危险状态。
- 主要道路、山脉、河流和关隘。
- 己方军团与已侦察到的敌军行动。
- 当前月份、季节、钱粮、威望和行动提醒。

### 7.2 交互

- 拖拽移动地图，滚轮缩放。
- 点击城池打开摘要；己方城池可进入城池管理，敌方城池直接提供“出征此城”并带入出征准备页。
- 点击军团查看主将、兵力、粮草、目的地和预计到达时间。
- 使用筛选器查看外交关系、粮食、民心、兵力或人才分布。
- 未完全侦察的敌城只显示情报区间，并注明情报更新时间。

## 8. 城池经营

城池独立管理、城市府库、城务额度、太守委任与减负规则详见 `docs/08-domestic-system-requirements.md`；若本节概述与专项文档冲突，以专项文档为准。

### 8.1 城池属性

| 属性 | 作用 |
| --- | --- |
| 人口 | 税收、征兵上限和劳动力来源 |
| 农业 | 季度粮食产出 |
| 商业 | 月度金钱产出 |
| 民心 | 人口增长、征兵效率和事件倾向 |
| 治安 | 盗贼、腐败和叛乱概率 |
| 城防 | 守城评分、城破后的设施损失 |
| 文化 | 搜索人才、教育和研究效果 |
| 疲敝 | 连续征兵、战争和灾害带来的负面状态 |

“初平群雄”发生于 194 年战乱期。城池人口表示能够直接登记、纳税和服役的人口，不等同于整个郡国的推算人口；33 城初始合计约 250 万，单城约 4.3–12.8 万。初始常备守军合计约 35.8 万，单城约 0.69–1.47 万，边城、州治和战略要地按等级保留差异。

### 8.2 行动

- 开垦：提高农业，需要金钱和负责武将。
- 经商：提高商业，可能触发商人事件。
- 巡查：提高治安和民心。
- 修城：提高城防并消耗材料。
- 征兵：消耗人口、金钱与民心。
- 训练：消耗粮食，提高部队训练度。
- 搜索：发现人才、宝物、情报或事件。
- 赈灾：消耗储备，降低灾害损失并提高民心。

每座城每月拥有有限行动槽。太守能力、城市规模和设施可以增加槽位或效率。

## 9. 武将系统

武将等级、五维成长、特性品质、名将专属特性、特殊部队适配、官阶与俸禄的详细规则见 `docs/09-officer-progression-system-requirements.md`；若本节概述与专项文档冲突，以专项文档为准。

### 9.1 基础能力

- 统率：带兵、守城和部队组织。
- 武力：冲阵、威慑与个人勇武。
- 智谋：计策、情报和识破能力。
- 政治：建设、税收、治理和交涉。
- 魅力：招募、民心、关系和士气。

数值范围为 1 至 100。战斗和内政不直接使用单项裸值，而通过职责公式组合。

### 9.2 人格与状态

- 性格标签：忠义、谨慎、豪勇、贪财、多疑、爱民、野心等。
- 忠诚：代表留在势力中的意愿，不等同于好感。
- 功勋：影响官职要求和派系声望。
- 关系：亲族、好友、敬佩、竞争、仇敌。
- 状态：健康、受伤、疲劳、被俘、在野、任职或出征。

### 9.3 招募

- 城内搜索发现。
- 俘虏劝降。
- 外交交换或事件加入。
- 仅敌方忠诚低于 60 时可进行策反；策反不获得额外成功率加成，但需要支付行动费用。

招募前根据目标与执行者实时显示预计成功率，但不向玩家展示内部计算公式；招募方式由目标状态自动决定，不能把在野登庸、俘虏劝降和敌将策反任意混用。

## 10. 官职与任命

本节中的太守、军师、主将等表示实际职务，不等同于官阶。文武官阶、晋升条件、条件属性、编制名额与月度俸禄详见 `docs/09-officer-progression-system-requirements.md`。

- 太守：管理城池，影响全部内政收益和地方稳定。
- 军师：提高情报、计策和战力预估精度。
- 主将：决定军团统率、阵型和主战术。
- 副将：补足能力、提供兵种或战术加成。
- 内政官：专门执行农业、商业、治安等任务。
- 外交官：执行结盟、交涉和交换。

同一武将不能同时出现在两个地点或职责中。调任需要时间，前线换将可能降低短期士气。

## 11. 外交

第一版包含：

- 关系值与态度：友好、中立、警惕、敌对。
- 停战、通商和俘虏交换；同盟与共同进攻暂不进入当前版本。
- 外交提案展示双方收益、AI 接受倾向和主要阻力。
- 违约会降低信用，影响所有势力对玩家的判断。

联姻、同盟、共同进攻、附庸、朝廷体系和复杂间谍网进入后续版本。

## 12. 出征与战斗评分

> 战斗专项的军团人数、全武将职责、分阶段公式、动画和分层攻城要求，详见 [战斗系统优化需求](07-battle-system-requirements.md)。

### 12.1 出征准备

玩家依次选择：

1. 目标城池或敌军。
2. 主将与最多两名副将。
3. 各兵种实际人数；它们的合计就是军团兵力，不再额外设置一次“总兵力”。
4. 携带粮草。
5. 行军路线。
6. 作战倾向：稳健、标准、激进。
7. 主战术；界面必须即时显示兵种条件是否满足。

界面必须即时显示：预计战力、补给天数、行军时间、胜率区间、主要优势和主要风险。

玩家可使用“全城出击”把出发城全部驻军按当前推荐阵型带走；主副将统兵值作为效率提示而非硬性上限。AI 仍必须遵守最低守军与风险策略，不能因此无脑清空城市。

### 12.2 战斗评分构成

双方总评分由以下部分构成：

- 基础兵力：实际兵力乘兵种基础系数。
- 将领：统率为主，武力、智谋和副将协同为辅。
- 部队：训练、士气、兵种克制和伤病。
- 后勤：粮草充足度、行军疲劳和补给线。
- 战场：守城、地形、天气和路线。
- 情报与战术：情报质量、战术匹配和计策结果。
- 波动：由存档种子产生的小幅确定性随机值。

战前不能显示敌方隐藏项的精确值，只能根据情报质量给出区间。

### 12.3 结果等级

| 评分比值（己方 / 敌方） | 结果倾向 |
| --- | --- |
| 1.45 以上 | 大胜 |
| 1.15 至 1.45 | 胜利 |
| 0.90 至 1.15 | 胶着，依据战术和波动判定 |
| 0.70 至 0.90 | 失败 |
| 0.70 以下 | 大败 |

比值只确定结果倾向，最终还需考虑撤退路线、城防是否被突破、粮草和特殊事件。

### 12.4 结算内容

- 胜负等级和双方最终评分。
- 双方损失、溃逃和伤兵。
- 武将受伤、被俘、立功或关系变化。
- 城防损失、城市是否易主。
- 缴获钱粮和物资。
- 影响胜负的前三项因素。
- 一段由结构化结果组合出的战报叙述。

同一存档、相同输入和相同随机种子必须得到相同结果，方便测试与复盘。

## 13. AI 势力

AI 每回合按以下顺序决策：

1. 识别生存威胁和资源缺口。
2. 设定一个主目标和最多两个辅助目标。
3. 分配内政、外交和军事预算。
4. 评估可执行行动并按效用选择。
5. 保留必要的守军和粮草，不进行明显自杀式出征。

AI 性格影响权重。扩张型更愿意承担风险，守成型优先稳定城池，权谋型更频繁使用外交和策反。

## 14. 事件系统

事件分为天下、城市、武将和战后四类。事件由条件、权重、冷却时间、选项和效果构成，尽量数据驱动。

选择应提供可预期的直接代价，长期或隐藏后果可以保留悬念。例如赈灾会明确消耗粮食并提高民心，但是否获得名士投奔可以作为后续事件。

## 15. 胜负条件

当前剧本胜利条件：控制 33 座城池中的至少 9 座，并连续维持 3 个月以触发“九城归心”；其余势力可选择归降或继续抵抗。

失败条件：失去全部城池时势力立即覆亡；君主死亡与继承规则在后续人物系统里补充。

## 16. 信息与可用性要求

- 任何资源变化都能查看来源和去向。
- 任何高风险操作都在确认前显示预计代价。
- 颜色不是唯一的信息手段，势力同时使用城池徽记、简称、文字标签和领地边界区分；领地底色不得铺设星号、斜线、波纹等重复符号。
- 关键文字在 1920×1080 下不小于 14 px。
- 支持降低动态效果，重要动画可跳过。
- 不使用生成图片中的文字作为正式信息。

## 17. 第一版不做

- 逐个表现士兵的选择、碰撞和精细寻路；实时操作以战斗队为最小单位，不扩展为逐兵控制。
- 武将单挑、舌战小游戏。
- 多人联机和赛季系统。
- 完整朝廷、官爵、后宫和家族生育系统。
- 数百城池和上千武将的大地图。
- 付费抽卡、体力或强制在线机制。

## 18. 验收标准

- 玩家可以完成从内政到占领敌城的完整闭环。
- 连续推进 24 个回合不会出现资源负数、武将重复或城市无归属。
- 战前预估与战后解释使用同一套评分来源。
- AI 能经营、守城和选择合理目标，且不会把所有兵力无脑派出。
- 保存后重新载入，所有城市、武将、军团、事件冷却和随机种子保持一致。
- 1366×768 至 2560×1440 下核心界面可用，无横向信息截断。

## 19. 完整增强需求（M6–M10）

本节由 2026-07-13 试玩反馈确定，属于后续持续开发的正式需求，不以 M0–M5 首版验收替代。

### 19.1 M6：人才与关系

- 建立亲族、好友、敬佩、竞争、仇敌五类双向或有向关系图，关系强度与历史变化进入存档。
- 城内搜索可以真正发现尚未登场的在野武将；发现与招募是两个明确阶段。
- 支持在野登庸、俘虏劝降、外交交换和对低忠诚敌将策反。
- 每次招募前必须随目标与执行者变化显示预计成功率，但不展示内部公式；执行后再公布最终随机判定。
- 增加武将详情页、势力人才总览和可交互关系网络；任何武将保持唯一势力、地点和状态。

### 19.2 M7：战争后果与行军

- 战斗确定性计算轻伤、重伤、恢复月份、俘虏、伤兵和溃逃兵。
- 胜方缴获金钱、粮草和军备物资，全部进入资源台账。
- 每场战斗结算后，胜方参战武将忠诚 +1，败方参战武将忠诚 -1；平局不变。被俘、释放、救援和共同作战可继续扩展关系影响。
- 主战术兵种条件不足时禁止出征或开战，并直接显示所缺兵种与比例，不进行自动替换。
- 城防未破而进攻方继续作战时进入多回合围城；记录围城月数、攻守补给和撤退条件。
- 支持多段道路选路；任意有可达路线的己方城市都可作为出发城，比较天数、地形、关隘和补给风险。
- 两支敌对军团在同一道路上的本回合行军区间相交时立即触发野战，不允许地图标记互相穿过后继续各自攻城。
- 玩家可在天下地图选中己方或敌方军团，从仍在途的己方军团中下达“改令拦截”；若本回合已经行动则保留军令至下回合执行，若仍可行动则立即推进并检查接战。

### 19.3 M8：AI 托管与外交

- 玩家可开启/关闭 AI 势力托管，并选择保守、均衡、扩张三种授权策略以及允许自动执行的内政、任命、外交、出征范围。
- 托管 AI 使用与普通 AI 相同的感知、预算、候选行动、效用评分和硬约束；所有自动决策可解释、可暂停、可审计。
- AI 主动向玩家提出停战、通商；提案进入待处理队列，不能静默生效。
- 同盟与共同进攻延后到外交格局扩展阶段，不在当前外交界面提供占位功能。
- 权谋型 AI 可对低忠诚敌将策反；玩家可查看主要成功因素与防范建议。
- 信用、撕约、援军失约和多次拒绝形成长期外交记忆；AI 界面展示完整候选比较和未行动原因。

### 19.4 M9：调任、情报、城池与灾害

- 武将跨城调任进入在途状态并消耗真实月份；前线换将产生临时士气影响。
- 敌军只有在侦察范围、道路接触或军师情报刷新后才显示位置；兵力精度随情报质量变化。
- 点击军团显示将领、副将、士气、训练、粮草、路线、战术和预计到达时间。
- 搜索或军师行动可刷新指定城市/道路情报，并记录来源和有效期。
- 工坊、瓮城、学宫、校场、常平仓等完整参与规则；设施支持升级、受损与修复。
- 旱灾、洪水、瘟疫、盗匪具有城市状态和持续时间；赈灾按类型产生不同成本与结果。
- 月度结算武将俸禄；支持城市府库与势力府库之间的受控转账并写入台账。

### 19.5 M10：长期局势与工程体验

- 君主死亡后判断合法继承人；无继承人时失败，有继承人时完成势力交接、忠诚波动和公告。
- 败亡势力根据关系、信用和剩余武将选择归降、流亡或继续抵抗。
- 控制 9 城时其余势力收到一次归降决策；记录势力兴亡、事件和历史战报年表。
- 完整战役结算展示胜负原因、回合数、城池、人才、战役和外交统计。

## 23.1 M14.1：AI 金粮平衡、零城覆灭与全局视角快捷键

- AI 不再只判断当月是否付得起行动费用；必须按城池数量、驻军规模和未来数月维护计算动态金粮安全线。

- 当粮储已经充足而金钱仍超过安全线五倍时，将超额金钱采购为军备，避免后期无限囤金。
- 金粮压力必须改变内政优先级：缺粮提高农业权重、缺金提高商业权重，资源危机时降低非经济建设优先级。
- 征兵、普通出征、共同进攻和策反均不得突破必要储备；最后据点进攻可以降低但不能取消军粮安全线。
- 全 AI 模式下原玩家势力使用同一金粮调剂规则；观战榜单直接展示各存续势力当前金钱与粮草。

- 势力失去最后一座城时立即覆灭，不因仍有在途军团、盟友或存活武将继续保留势力状态；残军解散，未被俘武将转为在野。
- 自动演进榜单将零城势力置于榜尾，综合战力归零，并显示“已覆灭”“0 城”“残军解散”等明确标识。

- 验收覆盖金多粮少、粮多金少、不套利、出征储备、零城覆灭、榜单排序及普通按钮焦点下的 W/A/S/D。

## 24. 第二阶段玩法深化目标（M15–M21）

M0–M13 已经建立“经营—人才—外交—战争—演进—结算”的完整系统，M14 正把剧本扩展到全国 16 方、33 城。但系统齐全、地图扩大不等于长局中持续有趣。第二阶段不继续横向堆叠资源种类和按钮，而是让已有系统互相施压，使玩家每几个月都要回答以下问题：

- 当前最重要的是恢复国力、争夺人才、稳住新领地，还是发动下一场战争？
- 哪些事情必须亲自决定，哪些城市和军团应交给可信的武将？
- 是集中资源完成一个明确战略，还是同时照顾多条战线并承担效率损失？
- 为了赢下一场战役，是否值得破坏关系、透支民心、背弃盟约或放弃其他方向？
- 版图扩大后，如何处理治理成本、地方派系和周边势力联合遏制，而不是只靠更多城池滚雪球？

### 24.1 深化原则

1. **增加决策，不增加流水账**：新增系统应优先形成互斥选择、委任和条件策略，不要求玩家逐城重复点击。
2. **扩张同时带来负担**：新城提供人口、位置和产出，也带来整合、驻防、粮道和行政压力。
3. **武将通过立场影响局势**：关系不能只停留在招募加成，必须进入议政、任命、协同和承诺兑现。
4. **战争是跨月计划**：单次出征仍用评分结算，但玩家应能组织目标、补给、增援、牵制和撤退。
5. **优势必须引发世界反应**：强大势力会遭到提防、联盟和有条件归附，而不是等弱小势力逐个送死。
6. **所有规则继续可解释**：关键结果展示来源、阈值和主要影响因素；AI 与玩家遵守相同资源和行动约束。

### 24.2 范围约束

- 保持“一回合一个月”和“无逐单位战场微操”的产品边界；玩家军团接战可进入独立演算画面。
- 不引入装备刷取、随机词条、卡牌抽取或需要反复练级的养成循环。
- 不把金、粮、威望、军备扩张为大量细碎物资；城市特产优先提供规则能力和贸易价值。
- 不要求玩家管理每支部队的每日行程；跨月计划以命令、条件和阶段结算表达。
- 新系统先覆盖现有 16 势力、33 城“初平群雄”剧本；内容型需求可以先在 2–4 个势力制作切片，但规则、AI 与存档从一开始就必须支持全部势力。
- 统一使用 Godot 客户端，提供无需开发服务器、可离线运行的 macOS 与 Windows 桌面版本。

## 26. M15：战略方略与行政

### 26.1 季度方略

- 每年一月、四月、七月、十月进入方略议定；玩家选择一个主方略和最多一个辅助方略，有效期三个月。
- 首批主方略包括：休养生息、富国强兵、广纳贤才、巩固新土、北定强敌、合纵连横。
- 方略不能只是全局数值加成，必须同时包含目标、允许投入和代价。例如“富国强兵”提高训练与军备效率，但增加金粮维护并降低民心恢复。
- 连续完成方略目标积累“国策进展”，失败不会直接惩罚资源，但会降低威望并影响提出该方略武将的信任。
- 方略界面展示建议者、主要收益、机会成本、三个月目标和当前完成度。

### 26.2 政令与亲自决策

- 势力每月拥有有限政令额度，用于跨城调度、改任、重大建设、主动外交、情报专项和战役建立。
- 城池原有行动槽继续约束当地执行能力；普通内政由已委任的太守自动完成时不消耗玩家政令。
- 玩家可以亲自覆盖托管安排，但需要消耗政令，并记录“为何打断原计划”。
- 政令额度主要由君主政治、军师能力、都城设施、势力规模和行政负担决定，不允许单纯随城池数线性增长。
- 政令不足时允许查看和规划，不允许静默透支；所有会消耗政令的操作在确认前展示剩余额度。

### 26.3 行政负担与区域委任

- 每座城市产生基础行政负担，距离都城远、新近占领、治安低、太守空缺和道路中断会增加负担。
- 行政能力由君主、军师、太守政治、学宫/驿站类设施和稳定道路提供；超过上限后按超出比例增加腐败、税粮损耗和事件风险。
- 玩家可建立不超过三个管理区域，指定治所、区域长官、目标和资源边界；区域不是新的地图层级，不改变城市归属。
- 区域长官可按“民生、建设、守备、征募”之一执行托管，并遵守最低库存、禁止征兵、不得拆借前线粮草等边界。
- 托管结果按月汇总为一张区域报告，异常情况单独提醒；玩家不需要逐城确认正常行动。

### 26.4 M15 验收

- 玩家能够在三个月内制定方略、委任一个区域并看见自动执行和目标进度。
- 同样拥有 3 城与 10 城时，玩家需要亲自处理的常规内政操作数量不会随城池数同比增长。
- 10 城势力若不任命太守、不维护道路且持续扩张，会出现可解释的行政损耗；完成委任和建设后能够恢复。
- AI 会选择与自身局势相符的方略和托管目标，并在审计中展示至少三个候选及未选原因。

## 27. M16：城市分工、区域网络与后勤

### 27.1 城市定位

- 每座城保留现有属性，并根据地理、设施和发展形成一个主要定位：粮仓、商埠、军镇、学府或枢纽。
- 定位由玩家选择建设方向后逐步形成，不在开局永久锁死；改换定位需要时间和成本。
- 定位提供能力而非单纯百分比：粮仓可建立战略储备，商埠可开通贸易，军镇可整补军团，学府可培养与发现人才，枢纽可降低区域调度成本。
- 每个定位至少有一个依赖其他城市的短板，鼓励区域协作而不是把所有城市建成同一模板。

### 27.2 道路、贸易与粮道

- 相邻己方或通商城市可以形成运输线，按月转运金粮并产生运输损耗；危险道路、灾害、敌军和关隘会改变损耗与中断概率。
- 军团出征时绑定一个补给源和一条补给链。补给不是创建军团时一次性扣除，而是携行粮与后方月度补给共同构成。
- 敌军可执行“袭扰粮道”，守方可执行“护送补给”；两者通过军团、道路控制、情报和武将能力评分结算。
- 补给中断先消耗携行粮，再降低士气、增加疲劳和溃兵；恢复道路后不会立即清除已产生的负面状态。
- 地图提供“区域网络”信息模式，区分畅通、拥堵、受威胁和中断，不显示未经侦察确认的敌方精确袭扰值。

### 27.3 新占领城市整合

- 城池易主后进入整合状态，初始整合度由原归属时间、民心、战斗破坏、双方关系和是否劫掠决定。
- 玩家在“安抚、军管、自治”三种政策中选择一种：安抚消耗金粮、恢复慢但民心高；军管短期稳定和征用强但长期反抗高；自治产出较低但行政负担小。
- 原势力武将、亲族和邻近敌军会影响整合事件；任命当地认可或爱民武将可提高整合效率。
- 整合完成前，城市不能提供满额税粮、征兵和行政能力，也不能作为无风险的深远补给源。
- 失去整合中的城市会影响威望和相关将领关系，避免反复易主完全没有后果。

### 27.4 M16 验收

- 同样数值的两座城市因定位和道路位置不同，能产生可感知的战略价值差异。
- 玩家能够建立一条跨三城补给链，遭到袭扰后从地图、军团详情、资源台账和战报中看到同一原因链。
- 新占城至少需要 3 个回合且一次明确政策选择才能恢复主要能力，不能在易主当月贡献满额产出和征兵。
- 60 回合压力测试中，大势力的平均行政与驻防成本随版图增长，不能仅凭城市数量形成无条件指数优势。

## 28. M17：武将成长、志向与议政

M17 中等级、阅历、五维成长、特性、官阶和俸禄的实现以 `docs/09-officer-progression-system-requirements.md` 为详细需求来源；本节继续定义志向、承诺、议政、派系和军团协同。

### 28.1 功勋成长

- 武将通过治理、外交、情报、战斗、救援和完成方略获得对应履历，不通过重复点击获得通用经验。
- 履历达到条件时获得一个专长选择，例如“屯田能手、守城宿将、长途奔袭、识人善任、纵横家”；专长改变可执行策略或条件，不大量叠加五维数值。
- 每名武将在一个剧本中最多获得两个后天专长，避免后期人人全能。
- 受伤、长期疲劳和年龄会影响可承担任务；恢复、休养和退居顾问均为可选安排，不用永久随机减值制造挫败。

### 28.2 志向与承诺

- 武将拥有一个主要志向：建功、治民、求名、求稳、复仇、兴族或匡扶理念；志向对玩家可见，但满足方式可以有多种。
- 招募时承诺的官职、共同进攻、释放亲友和重用建议进入承诺列表，包含期限、进度与违约预警。
- 满足志向提高忠诚和主动提案质量；长期压制、失信或明显任人唯亲会降低忠诚，并可能形成请求、请辞、倒向派系或出走事件。
- 负面结果必须至少提前一个月给出迹象，除明确的敌方策反外，不允许无提示地让高忠诚核心武将突然叛逃。

### 28.3 议政与派系

- 每次季度方略前召开一次议政，2–4 名有资格武将基于性格、关系、职务和局势提出不同方案。
- 采纳某人建议会增加其信任与功勋，也可能引发竞争者不满；拒绝不会固定扣忠诚，只有涉及志向、承诺或连续忽视时才累积影响。
- 三名以上关系紧密且诉求相近的武将可形成松散派系。派系不提供永久阵营标签，会随任命、战功、婚亲和关系变化重组。
- 派系影响建议、推荐人选、危机支持和继承，但不能直接夺取玩家控制权；重大内讧必须通过事件和可干预选择发展。

### 28.4 军团协同

- 主副将关系、共同作战次数、战术偏好和竞争状态形成协同值。
- 高协同提高战术执行、撤退秩序和增援成功率；仇敌或争功可能降低协同，但高统率主将和明确军令可以压制影响。
- 战后按真实行为改变关系，例如救援、抢功、违令追击、共同守城和放弃友军；关系变化必须进入战报。

### 28.5 M17 验收

- 任意核心武将在 24 回合中可通过至少两种履历路线形成不同专长，而不是只提升同一组数值。
- 招募承诺、官职承诺和共同进攻承诺均有期限、提醒、兑现与违约后果，并能完整存档。
- 同一次季度议政至少出现两个立场不同且原因可解释的提案；采纳结果会反馈到方略、关系和后续事件。
- 军团协同对战前预估和战后因素使用同一来源，相同存档与命令保持确定性。

## 29. M18：兵种编制、道路行军与战役协同

### 29.1 兵种体系与混合编制

- 保留步兵、枪兵、弓兵、骑兵、攻城兵五个基础兵种，但每个兵种必须拥有明确职责、克制对象、被克制对象、地形偏好、行军速度、粮草消耗和军备成本，不能只体现为一个基础战力系数。
- 基础克制关系如下：枪兵强力拒止骑兵；步兵不克制骑兵；骑兵克制步兵和弓兵并擅长追击溃军；弓兵压制缺少掩护的步兵、枪兵；攻城兵强于破坏城防但在野战中需要护卫。
- 克制不是固定胜负。兵力比例、统率、训练、士气、阵型、地形、天气、疲劳和战术可以放大、削弱或反转克制结果，单项克制倍率必须设置上下限。
- 军团从“单一兵种＋总兵力”升级为兵种编制。每支军团选择一个主力兵种和最多两个辅助兵种，攻城兵占用一个辅助位置；各兵种人数之和直接派生军团总兵力，界面不得再提供第二个总兵力输入。
- 混合编制形成协同：步/枪掩护弓兵、骑兵侧击、步骑追击、步兵护卫攻城兵。协同要求达到最低兵力比例，并受主副将专长与战术影响；不能只放 1 名兵种就获得完整加成。
- 骑兵和攻城兵需要更多军备与适合的城市设施/地区条件；不新增大量细碎资源，继续从军备、设施、城市定位和区域特性表达组建门槛。
- 伤亡、伤兵、溃兵、补员和缴获必须按兵种分别记录，不能在战后重新任意改变幸存部队构成。
- 出征界面同步展示敌方已知编制、克制链、协同是否生效、地形影响、速度瓶颈和粮草消耗；情报不足时只显示兵种倾向或区间。

### 29.2 固定道路与真实行军距离

- 城池之间只能沿剧本定义的固定道路行军，不支持脱离道路的直线移动。每条道路记录连接城池、长度/基础日数、地形、关隘、道路等级和可容纳军团规模。
- 出征必须选择一条由连续道路组成的合法路线。路线总距离等于各路段距离之和，预计回合数以军团每回合行进力计算，再叠加最慢兵种、季节、天气、疲劳、关隘和道路拥堵修正。
- 军团拥有“每回合基础行进距离”。当前首版以步兵 18 道路里程为基准；纯骑兵 21、步兵 18、枪兵 17、弓兵 16，编入攻城兵后降为 12；混编军团取最慢兵种作为速度瓶颈。数值应由统一规则计算，不允许 UI、玩家和 AI 各自维护常量。
- 军团记录当前道路、行进方向、路段进度、剩余距离、预计抵达回合和后续路段；玩家确认出征后立即执行本回合行军，后续回合也由玩家在出征页执行一次“本回合行军”，未抵达则停留在道路上的确定位置，不允许直接从出发城跳到目标城。玩家军团不再等到月末遮罩期间推进，月末仅自动推进 AI 军团。
- 玩家出征或继续行军后必须先回到天下地图，镜头聚焦军团并播放从原位置到本回合目标位置的移动；抵达并触发战斗时，行军表现结束后再打开战前军议，确保军团移动不被结算或战斗浮层遮挡。
- 全国道路距离按“近邻通常需要 1–2 回合、跨区域需要更多回合”的节奏配置。首版在原道路基线上统一放大为 2 倍，后续可按地形和试玩数据逐路调整；任何直连道路都不得因默认步兵速度而无条件一回合瞬间跨越。
- 出征确认界面必须显示路线总距离、军团每回合行进距离和预计抵达回合；地图上的在途军团必须同时显示当前位置与“本回合最多可到达位置”，让玩家在结束回合前判断截击、会合和抵达时机。
- 多条合法路线同时存在时，界面比较总日数、粮耗、地形、敌情、关隘和被截击风险；最短路线不一定是最安全路线。
- 未获军事通行时不能穿越第三方领地；目标城易主、道路中断或条约变化后，军团根据条件军令改道、等待或撤退，不能穿过无效路线。
- 地图上的军团位置必须由同一份路段进度计算，存档、读档和不同缩放下保持一致。

### 29.3 敌情发现与路途迎击

- 势力只有通过本方城市、道路哨探、军团侦察、盟友情报或军师行动发现敌军后，才能对该军团作出精确迎击决策；低质量情报只提供方向、规模区间和可能路线。
- 发现敌军出兵后，AI 必须生成并比较“固守目标城、出城迎击、封锁关隘、尾随观察、增援友城、避战保存实力”候选，不允许固定选择迎击。
- 迎击候选至少考虑：能否在敌军通过前到达同一路段、双方兵种克制、地形、主将、兵力区间、出发城最低守军、粮草、其他威胁和条约约束。
- AI 选择迎击时创建真实军团、扣除真实兵力和粮草并占用真实武将；不能在结算瞬间生成部队，也不能读取尚未侦察到的玩家隐藏路线。
- 两支敌对军团在同一道路、同一关隘或相邻行军区间发生时间重叠时，触发野外交战。战场地形取当前道路，攻守身份由迎击、行军方向、情报和先到达时间决定。
- 迎击军到达过早可选择设伏或占据关隘；到达过晚则转为追击、增援目标城或取消行动，并在战报解释未能接战的原因。
- 玩家享有与 AI 相同的敌情提醒和迎击入口，可以从附近城市或在途军团选择拦截点，并在确认前查看预计相遇日和到达区间。
- 多支军团争夺同一路段时按预计到达时间和军令优先级顺序结算，禁止同一军团在一个阶段参加两个地点的战斗。

### 29.4 战役目标

- 玩家可以建立一个持续多月的战役计划，选择目标城或区域、计划期限、主攻方向和最多两个辅助目标。
- 战役建立时必须指定主帅、可用军团、补给源、集结点和停止条件；未满足条件时允许保留计划而不强行出兵。
- 首批辅助目标包括：牵制邻城、封锁关隘、护送补给、截击援军和佯攻吸引守军。
- 一个军团同一时间只能属于一个战役；改派需要考虑距离、军令延迟和原计划损失。

### 29.5 条件军令

- 玩家为军团设置“推进、待援、固守、袭扰、撤退”姿态，并可添加一个条件，例如“补给低于 2 月撤退”“敌援军超过 8000 时待援”。
- 条件军令在回合阶段自动判定，触发与未触发原因都进入战役日志。
- 玩家每月可修改尚未执行的命令；军团正在交战、失联或道路被切断时，修改可能延迟一个月生效。
- AI 使用同一条件系统，不允许在结算时读取玩家隐藏命令后即时改变策略。

### 29.6 接战、增援与撤退

- 同一道路、关隘或目标周边的敌对军团可发生野外交战、截击或增援，不必等到攻城才结算。
- 增援是否及时到达由路线、集结时间、情报、主将统率和敌方牵制共同决定；战前只展示基于现有情报的到达区间。
- 撤退需要合法道路和接收城市；被截断时会增加溃兵、被俘和军备损失，不允许军团瞬移回城。
- 围城方可选择强攻、围困、撤围或围点打援；守方可坚守、突围、等待援军或有条件议和。
- 战斗仍使用结构化评分；玩家军团参与时进入敌军在左、我军在右的高质量 2.5D 大战场，按地形与地域切换正式战场原画，横向兵种动画按阵营相向播放；不开放逐个士兵的战场操作，指挥单位仍为战斗队。

### 29.7 战役复盘

- 战役结束后按月份展示目标变化、军团路线、关键接战、补给状态、命令触发和城市易主。
- 复盘列出计划完成度、主要成功因素、主要失误、资源总成本和武将关系变化。
- 玩家可以从复盘直接查看对应战报、资源台账和历史年表记录。

### 29.8 M18 验收

- 五个基础兵种均可实际招募、混编、补员和结算；至少覆盖枪强克骑、骑兵克步兵、骑兵突弓、弓压无掩护步枪、步兵保护后排和攻城兵破城六组可解释规则。
- 不存在跨平原、山地、守城三类场景都稳定占优的单一编制；同等资源下至少有三种有效配兵方案。
- 同一出发城和目标城选择不同固定路线时，行军日数、粮耗、地形和遭遇风险会真实变化；军团位置可在路段中保存和恢复。
- 步兵基准军团在 34 距离的直连道路上，第一回合只能前进 18 并停留在途中，第二回合才抵达；地图预览位置与回合结算后的实际位置一致。
- 不同编制使用同一路线时，预计回合数必须体现速度瓶颈；攻城混编不得沿用骑兵速度，玩家与 AI 的实际推进距离必须与出征预估一致。
- AI 看到可确认的敌军出兵后，会在守城与迎击等候选中进行比较；只有能及时到达、保留守军且收益合理时才创建迎击军团。
- 自动化验收必须覆盖“发现敌军—创建迎击—双方在道路相遇—野战结算—胜方继续/败方撤退”的完整流程，以及情报不足、迎击迟到和没有可用武将时不凭空接战。
- 玩家能组织至少两个己方军团完成一次“主攻＋护粮/牵制”的三个月战役，而不是分别点击互不关联的出征。
- 补给线被切断、援军迟到和主动撤退均有地图过程、评分影响和可解释结算。
- AI 能建立、调整和终止战役，不会在目标已经失效或补给持续不足时无限添兵。
- 同一战役相同输入和种子产生相同的命令触发、接战顺序和战役复盘。

## 30. M19：外交格局与天下反应

### 30.1 威胁与共同利益

- 每个势力对其他势力分别计算威胁感知，来源包括邻接版图、近期扩张、军力差、违约、关隘控制和共同敌人。
- 威胁不是公开的精确数值；外交界面显示低、中、高等级、最近变化和已知主要原因。
- 势力快速扩张会推动弱方改善关系、互通情报、建立防御联盟或向第三方求援，但宿敌、低信用和利益冲突会阻止无条件抱团。
- AI 不因玩家身份获得额外仇恨，自动演进和玩家局使用同一判断。

### 30.2 复合议约

- 一次交涉可以组合主条款和最多两个附加条款，主条款包括停战、通商、同盟、共同进攻、军事通行、保护和臣属。
- 附加条款包括期限、金粮、俘虏、目标城、最低出兵、道路开放和违约赔偿。
- 界面分别展示每一条款对双方效用的影响，不能只显示一个总成功率。
- 提案、还价、接受和拒绝均消耗时间；紧急停战可立即生效，但需要更高让步。

### 30.3 臣属、联盟与战争目标

- 弱势力可选择臣属以保留城市和武将自治，宗主获得贡赋、通行和对外战争义务，同时承担保护责任。
- 臣属不是永久吞并；信用下降、保护失败或实力逆转会增加脱离倾向，宗主可选择安抚、改约或武力压制。
- 宣战需要选择公开目标：夺取城市、救援盟友、惩罚违约、迫使臣属或生存防卫。目标影响威望、停战条件和其他势力态度。
- 无目标的反复宣战、撕毁停战和背刺盟友会积累“失信”记忆，短期收益不能由送礼立即洗清。

### 30.4 M19 验收

- 当任一势力在 12 个月内连续吞并三城时，邻近势力会基于各自利益作出至少两种不同反应，而不是统一宣战。
- 玩家能够完成一次包含停战、俘虏交换和粮食补偿的复合议约，并在界面看懂每项接受阻力。
- 臣属关系包含贡赋、保护、脱离和转为吞并的完整闭环，且双方 AI 都能使用。
- 外交记忆能追溯到具体条约和战役，存档后不会丢失或重复结算。

## 31. M20：势力志业、连续事件与多路径终局

### 31.1 势力志业

- 16 个势力各拥有一条三阶段志业，体现其开局处境和君主性格，而不是提供无条件永久加成。
- 每阶段至少有两种完成方式。例如“安定徐州”可以通过控制关键城市完成，也可以通过同盟、民心和停战维持完成。
- 完成志业解锁势力策略、人才事件或外交身份；失败不会锁死整局，但会改变后续目标和事件分支。
- 君主更替后重新评估未完成志业，继承人可以延续、修正或放弃旧路线，并承担相应关系影响。

### 31.2 连续事件与天下议题

- 事件支持 2–5 个阶段、触发窗口、参与实体、分支状态和收束条件；前次选择必须实际改变后续可用选项。
- 首批连续事件覆盖：灾后流民安置、名士举荐、边境争端、将领不和、盟友求援、继承争议和地方豪强归附。
- 每 12–24 个月出现一个天下议题，例如大灾、朝廷诏令、关中饥荒或名士品评，持续影响多个势力并形成争夺焦点。
- 历史人物和地名提供题材感，但事件不强迫局势照历史演出；已败亡角色或已易主城市不能继续触发不合时宜的固定剧情。
- 事件结算进入个人列传、势力年表和天下史书三个层级，支持按人物、城市、战役和条约筛选。

### 31.3 全国剧本的终局阶段

- 启用 M20 后，第 15 节与 M11 中“控制九城并维持三个月”的旧规则不再直接结算全国剧本；控制至少 15 城并维持三个月才触发“问鼎天下”。
- 问鼎后选择一条公开终局路线：
  - **霸业**：控制至少 22 城，并使其余有城势力败亡、臣属或接受最终停战。
  - **王道**：控制至少 15 城，平均民心、威望和信用达到要求，并获得至少五个非敌对势力承认。
  - **盟主**：领导有效联盟连续 12 个月，联盟合计控制至少 24 城，并完成一个共同天下议题。
- 选择终局路线会改变其他势力态度和后续事件；允许付出威望和关系代价改换一次路线，不允许无成本来回切换。
- 全 AI 自动演进使用同样的终局路线判断；达到回合上限时才使用综合实力排名作为阶段胜者。

### 31.4 难度与复玩设置

- 三档基础难度继续保留，并拆解为可见参数：AI 风险偏好、补给严厉度、行政负担、情报模糊度和事件压力。
- 自定义设置只调整规则参数，不直接给 AI 免费资源、无视粮道或读取隐藏信息。
- 新游戏可选择“势力志业引导”或“纯沙盘”；纯沙盘关闭专属目标奖励，但保留关系、行政、战役和外交规则。
- 结算比较不同路线下的城池、民心、人才、战损、信用、履约和治理成本，不只统计最终总兵力。

### 31.5 M20 验收

- 至少 4 个势力拥有可完整游玩的三阶段志业，其余势力在同一版本上线前补齐，不允许只给玩家默认势力制作内容。
- 至少 12 条连续事件可跨 3 个以上回合保留分支状态，且不会因角色死亡、势力败亡或城市易主产生逻辑矛盾。
- 霸业、王道、盟主三条终局路线均可由玩家和 AI 实际达成，并有互斥条件和不同世界反应。
- 同一势力以不同方略、志业分支和终局路线完成两局时，关键年表、主要盟友与资源曲线应产生明显差异。

## 32. 第二阶段实施顺序与总验收

### 32.1 推荐切片

1. **M15 最小切片**：季度方略、政令、一个管理区域和月度托管报告。先解决多城操作膨胀。
2. **M16 最小切片**：两种城市定位、真实补给源、占领整合。先让地图位置和扩张成本产生差异。
3. **M17 最小切片**：志向承诺、季度议政、一个后天专长。先让武将关系进入持续决策。
4. **M18 最小切片**：完成五兵种混编、至少两条不同道路路线，以及一次“发现敌军—真实出兵—道路相遇—野战结算”；闭环稳定后再扩展跨月战役与条件军令。
5. **M19 最小切片**：威胁感知、复合停战和保护/臣属。建立强者扩张后的世界反应。
6. **M20 内容切片**：先完成两个势力志业、四条连续事件和霸业终局，再扩展至完整规模。
7. **M21 桌面切片**：先用同一前端产物生成可离线启动的 macOS `.app` 与 Windows `.exe` 开发包，再补签名、安装升级和发布流程。

每个切片必须先形成可从 UI 完成、可读档恢复、AI 可使用的闭环，再扩充数量。不能先一次性录入大量事件和数值，最后才验证规则是否有趣。

### 32.2 第二阶段总验收

- 玩家控制 8 城连续运行 24 回合时，每月必须亲自处理的常规操作中位数不超过控制 3 城时的 1.5 倍。
- 至少存在三种可行发展路线，不能让“每月征兵并攻击最近弱城”成为所有势力和难度下的稳定最优解。
- 新占领、行政超载、外交威胁和人才诉求能够共同约束扩张，但都提供至少两种可行应对方式。
- 一场持续三个月以上的战役可以同时影响补给、城市整合、武将关系、外交态度和天下年表，且原因链可追溯。
- 兵种编制、固定道路、实际路段进度和途中迎击使用同一套玩家/AI 规则，战前预估与战后结算来源一致。
- 玩家与 AI 在政令、行政、补给、承诺、战役和外交上遵守相同规则；难度差异不得通过隐藏作弊实现。
- 120 回合确定性压力测试继续保持资源非负、实体唯一、引用有效，并新增行政、补给、承诺、战役和臣属不变量。
- 至少完成两轮 60 回合真人试玩记录：统计每回合决策耗时、重复操作数、方略选择、战争原因、终局路线和中途放弃原因，再据此调整数值。
- macOS 与 Windows 桌面版使用同一套 Godot/C# 领域规则与存档格式，相同存档输入得到相同的回合和战斗结果。

## 33. M15 开发前置：代码治理与数据一致性

本节来自 2026-07-13 代码评审。以下条目不是可推迟的“代码美化”，而是 M15–M21 扩展前必须建立的规则安全线。修复不得改变已经确认的玩法结果，确需调整数值时必须在实现文档记录迁移和新基线。

### 33.1 ENG-001：城市与势力府库守恒

- 明确税金、粮食从城市产出到城市府库、势力府库和维护支出的唯一流向，禁止同一笔月度收入同时完整增加两个账户。
- 城市与势力府库之间的转账保持总额守恒；征税、运输损耗、俸禄、维护和灾害分别进入资源台账。
- 增加单城、多城、转账前后、季度收成、零粮和读档后的总资产守恒测试。

### 33.2 ENG-002：完整存档运行时校验

- 为当前 `GameRuntimeState` 建立完整运行时 Schema，不使用只校验少量字段后整体放行的方式代替完整校验。
- 校验城市、势力、武将、军团、道路、外交、事件、AI、随机状态和新增里程碑字段，并检查跨实体引用和武将唯一位置。
- 读档顺序固定为：复制原记录、校验记录外壳与校验摘要、按版本顺序迁移、完整 Schema 校验、领域不变量校验、成功后替换当前游戏。
- 非法当前版本、迁移中断和引用损坏必须保留当前游戏并给出可处理错误，不能载入半份状态。

### 33.3 ENG-003：剧本数据完整性

- 势力、城市、道路、关隘、武将、战术和事件 ID 必须分别唯一；重复 ID 在数据加载阶段直接拒绝。
- 每个势力必须恰好拥有一名存活君主；君主姓名、所属、位置和任命与势力档案一致，不允许用空字符串填补缺失君主。
- 道路必须引用有效且不同的起止城市；关隘必须绑定有效道路；全国道路网连通性和孤城检查进入自动测试。
- 数据校验失败时输出具体实体 ID 和字段路径，方便定位内容错误。

### 33.4 ENG-004：AI 与势力内容数据驱动

- 16 个势力的 AI 性格、内政权重、风险倾向、兵种偏好和特殊策略通过稳定配置或标签表达，不在领域流程中按势力 ID 堆叠条件分支。
- 策反、骑战、守境、经商等能力由策略标签和通用规则解释；新增势力不需要修改通用 AI 执行器。
- 默认性格只能作为数据缺失保护，正式剧本不得让多个势力以错误的统一名称和完全相同权重运行。

### 33.5 ENG-005：模块规模与职责拆分

- `execute-game-command` 按城市、设施、人才、外交、出征和回合编排拆为领域命令处理器；顶层只负责穷尽分发，不保留具体业务公式。
- 新建独立 `domain/turn`，承载移动、灾害、建设、经济、AI、外交、事件、胜负和存档前快照的阶段流水线。
- 地图拆分为领地、道路、关隘、军团、城市标记和侧栏组件；几何/路线查询在模型层预先索引，不在每个渲染循环中反复线性查找。
- 战斗移动、核心 E2E 和其他超过代码规范审查上限的文件按职责拆分；数据表可以例外，但必须按领域分组并保持可审查格式。

### 33.6 ENG-006：自动代码规范门禁

- 统一 C# 格式、可空类型、导入顺序、未使用代码和 Godot 节点生命周期规则。
- 增加 `lint`、`format:check` 和统一 `check` 命令；完整门禁至少包含格式、Lint、类型、单元测试、端到端测试和生产构建。
- 禁止生产代码使用 `any`、直接 `Math.random()`、未说明的非空断言和 domain 对 app/features/browser API 的反向依赖。

### 33.7 工程治理验收

- 首月经济结算后，城市府库＋势力府库的总变化与台账净额完全一致，转账前后总额不变。
- 缺失任一关键字段、重复实体 ID、缺失君主、无效军团路线和武将重复位置的输入均被自动拒绝。
- 16 方全部读取明确 AI 性格配置，正式剧本中没有使用匿名默认势力名称。
- 超过 `03-code-standards.md` 审查上限的业务文件均已拆分，或在实现文档记录不可拆分理由与负责人确认。
- `./tools/run-godot.sh check` 在干净环境一次通过，并成为 M15–M21 每个切片的提交门禁。

## 34. M21：macOS 与 Windows 桌面发行

### 34.1 产品形态

- 使用 Godot 4.7 .NET 直接构建桌面游戏，macOS 与 Windows 复用同一套 C# 领域规则、场景和视觉资源。
- macOS 提供可直接启动的 `.app`，发布时可同时提供 `.dmg`；Windows 提供可安装的 `.exe`，可选提供便携版或 MSI，但 `.exe` 安装包是必交付物。
- 首批正式支持 macOS Apple Silicon、macOS Intel/通用包和 Windows 10/11 x64；其他架构不进入首个桌面验收范围。
- 桌面版完全离线可玩，不启动本地开发服务器，不依赖账号登录和外部 API。

### 34.2 窗口与系统体验

- 首次启动使用 1440×900 默认窗口，支持最大化、全屏和恢复上次窗口大小；窗口不得小于核心界面的最低可用尺寸。
- 应用名称、版本、图标、包标识和版权信息在 macOS、Windows 与“关于”界面一致。
- 支持标准退出、系统快捷键和关闭确认；自动演进或正在写存档时退出必须先完成原子保存或明确提示。
- 正式桌面版不显示编辑器或调试控制台；开发构建可以通过显式开关启用调试工具。

### 34.3 存档、升级与兼容

- 桌面存档写入系统规定的应用数据目录，不能依赖临时目录；自动档、手动档、设置和日志分开存放。
- 提供存档导出与导入，使 macOS 和 Windows 能交换同版本存档；导入前执行完整校验和版本迁移。
- 覆盖安装和小版本升级必须保留存档与设置；迁移失败时保留原文件和可读错误，不覆盖唯一可用存档。
- 崩溃或强制退出后，下次启动优先恢复最近成功写入的存档，不加载临时记录或半写入文件。

### 34.4 构建与发布

- 开发环境提供一条命令生成当前平台安装产物，发布流程能分别产出 macOS 和 Windows 文件并记录版本、提交和校验值。
- 开发测试包允许未签名；对外发布前支持 macOS 代码签名与公证、Windows 代码签名。证书属于发布凭据，不进入仓库。
- 构建产物不得包含源映射、测试报告、开发服务器配置、私钥或无关源资产；许可证和第三方素材授权清单随版本归档。
- 桌面壳权限遵循最小化原则，只开放存档读写、窗口和必要系统能力，不提供任意命令执行或不受控文件访问。

### 34.5 M21 验收

- 在全新 macOS 和 Windows 环境中安装/启动游戏，无需 Node.js、浏览器或网络即可开始新游戏。
- 完成“开始游戏—推进三回合—手动存档—退出应用—重新启动—读档继续”的真实安装包测试。
- macOS 与 Windows 导出的存档可以相互导入，反向导出后仍能通过当前数据校验与确定性结算测试。
- 覆盖安装新版本后存档、设置和随机游标保持一致；故意提供损坏存档时应用拒绝载入且不会影响其他存档。
- macOS `.app` 与 Windows `.exe` 的版本号、图标、应用名和结算结果一致；两端 60 回合压力测试得到相同结果。
